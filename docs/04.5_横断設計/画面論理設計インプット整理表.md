## 1. 目的

本ドキュメントは、画面論理設計に着手するための前提情報を整理し、

画面ごとの役割・利用者・入出力・依存API・依存データを明確化することを目的とする。

本整理により、以下を後続工程へ引き継ぐ。

- 画面一覧
- 画面遷移設計
- 画面項目定義
- ワイヤーフレーム
- APIとの対応
- データ供給前提との整合

---

## 2. 基本方針

```
・画面は「ユーザーの目的単位」で整理する
・画面ごとに主利用者を明確化する
・画面が必要とするデータは、正本 / 派生 / 集約の区分まで意識する
・画面の責務とAPIの責務を分離する
・管理画面 / 評価画面 / 一般ユーザー画面を分けて整理する
```

## 4. 画面区分の定義

| 区分               | 説明                                   |
| ------------------ | -------------------------------------- |
| 一般ユーザー画面   | エンドユーザーが推薦を受けるための画面 |
| 管理画面           | 運用・設定・監視・調査のための画面     |
| 評価画面           | human_eval を行うための画面            |
| ダッシュボード画面 | KPI / モデル / 技術監視の可視化画面    |

---

## 5. 依存データの読み方

| 区分       | 意味                                    |
| ---------- | --------------------------------------- |
| 正本       | recommendation_run 等の業務上の真の状態 |
| 管理正本   | semantic_config 等の設定・version       |
| ログ正本   | run_event_log 等のappend-only記録       |
| 派生データ | 集約・評価・監視用に生成されたデータ    |
| 集約データ | 日次・時間単位に集約した参照向けデータ  |

## 7. 画面設計上の重要前提

### 7.1 一般ユーザー画面

```
・入力は段階的に行う
・結果画面では推薦理由の説明を表示する
・評価を強制せず、自然なinteraction取得を優先する
```

### 7.2 管理画面

```
・run単位で追跡可能にする
・version比較ができること
・技術監視 / 業務監視 / モデル監視は分離する
```

### 7.3 評価画面

```
・human_evalは recommendation_result を基準に行う
・評価結果は改善ループへ接続する
```

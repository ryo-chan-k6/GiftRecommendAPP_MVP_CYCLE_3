## 1. 目的

本ドキュメントは、本サービスの実行基盤構成を定義し、以下を明確化することを目的とする。

- Web / API / Reco / Batch / DB / Cache の配置方針
- 各コンポーネントの責務
- 通信経路と接続方針
- 非機能要件を満たすための基盤上の前提
- MVP構成と将来拡張構成

## 3. 基盤構成の基本方針

### 3.1 基本方針

```
・MVPでは最小構成で開始する
・ただし API / Reco は初期から分離する
・正本データは PostgreSQL に集約する
・低レイテンシ参照は Redis を併用する
・重い生成処理は Batch に分離する
・Observability は基盤監視だけでなく業務・モデル監視まで含む
```

### 3.2 構成設計原則

```
・責務の異なるものは基盤上も分ける
・同期処理と非同期処理を分離する
・ステートレスなコンポーネントは水平スケール可能にする
・状態を持つものは一元管理する
・障害時の影響範囲を閉じ込める
```

## 5. コンポーネント別設計

## 5.1 Webアプリ基盤

### 役割

- 画面表示
- 入力UI
- JWT保持
- API呼び出し

### 方針

| 項目       | 方針                        |
| ---------- | --------------------------- |
| 実行形態   | Webホスティング基盤上で実行 |
| 状態       | ブラウザ側にUI状態を保持    |
| サーバ状態 | 持たない                    |
| 通信先     | APIサービス                 |

### 補足

- Webサーバは原則ステートレス
- 真のデータは保持しない
- ブラウザ上の状態は UX のための一時状態とする

---

## 5.2 APIサービス基盤

### 役割

- 外部公開API
- 認証 / 認可
- recommendation_run / result / interaction 等の保存
- Recoサービス呼び出し
- 管理API / 評価APIの制御

### 方針

| 項目     | 方針                           |
| -------- | ------------------------------ |
| 実行形態 | アプリ実行基盤上の独立サービス |
| 状態     | ステートレス                   |
| スケール | 将来水平スケール               |
| 接続先   | PostgreSQL / Redis / Reco      |
| 認証     | JWT Bearer                     |

### 基盤要件

- ヘルスチェックエンドポイントを持つ
- requestId を付与できる
- runId と紐づくログを出力できる

---

## 5.3 Recoサービス基盤

### 役割

- 入力解析
- User Meaning推定
- Retrieval
- Ranking
- 推薦結果生成

### 方針

| 項目     | 方針                               |
| -------- | ---------------------------------- |
| 実行形態 | APIとは独立したサービス            |
| 状態     | ステートレス                       |
| スケール | 将来水平スケール                   |
| 接続先   | PostgreSQL / Redis / Embedding API |
| 外部公開 | しない                             |

### 設計上の要点

- APIと分離するが、Reco内部を細かく別サービス化しない
- Reco内部はモジュール分割で疎結合化する
- 内部APIは少数に保つ

---

## 5.4 PostgreSQL 基盤

### 役割

- 業務正本
- 管理正本
- 派生データ
- 集約データ
- 監視用統計の永続保存

### 方針

| 項目 | MVP                   | 将来                         |
| ---- | --------------------- | ---------------------------- |
| 製品 | PostgreSQL (Neon想定) | PostgreSQL Primary / Replica |
| 構成 | 単一                  | 読み取り分離                 |
| 用途 | 正本 + 集約           | 正本中心、分析系分離可       |

### 保存対象

- recommendation_run / result
- interaction_log
- semantic_config / model_version
- normalization_stats
- item_feature
- item_embedding
- offline_eval_result
- feature_distribution_summary など

### 補足

- item_embedding は派生データだが、生成コストの観点から DB 保存対象とする
- user_embedding は DB 永続化対象としない

---

## 5.5 Redis 基盤

### 役割

- item_embedding キャッシュ
- user_embedding 短期キャッシュ
- retrieval系短期キャッシュ
- 一時的な高速参照

### 方針

| 項目   | 方針                        |
| ------ | --------------------------- |
| 用途   | 高速参照専用                |
| 正本性 | なし                        |
| 障害時 | DB / 再計算へフォールバック |
| TTL    | データ種別ごとに設定        |

### 主な対象

| 対象                 | TTL         |
| -------------------- | ----------- |
| user_embedding       | 5〜15分     |
| item_embedding cache | 数時間〜1日 |
| retrieval結果        | 数分        |

### 補足

- Redisは必須依存にしない
- cache miss でも業務継続可能とする

---

## 5.6 Batch基盤

### 役割

- 商品データ収集
- item_feature生成
- item_embedding生成
- normalization_stats生成
- 各種集約
- offline_eval / FB分類

### 方針

| 項目     | 方針                              |
| -------- | --------------------------------- |
| 実行形態 | 定期実行ジョブ                    |
| 実行方式 | MVPは単純実行、将来はワーカー分散 |
| 冪等性   | 必須                              |
| 再実行性 | 必須                              |

### ジョブ分類

| ジョブ      | 内容                                |
| ----------- | ----------------------------------- |
| ETL         | 商品データ取得・整備                |
| Feature     | item_feature生成                    |
| Embedding   | item_embedding生成                  |
| Stats       | μ/σ・分布統計                       |
| Aggregation | interaction / metric / quality 集約 |
| Evaluation  | offline_eval / FB分類               |

---

## 5.7 監視基盤

### 役割

- システム監視
- アプリ監視
- 業務監視
- モデル監視

### 方針

| 項目     | 方針                                       |

| 監視対象 | System / App / Business / Model            |

## 6. 通信経路設計

## 6.1 OL通信経路

```
Browser
→ Web
→ API
→ Reco
→ PostgreSQL / Redis
→ Reco
→ API
→ Browser
```

### 方針

- 外部公開は API のみ
- Reco は内部通信のみ
- API / Reco 間も認証対象

---

## 6.2 BT通信経路

```
外部商品データ
→ ETL Batch
→ PostgreSQL
→ Feature Batch
→ PostgreSQL
→ Embedding Batch
→ Embedding API
→ PostgreSQL / Redis
→ Aggregation Batch
→ PostgreSQL
```

## 10. MVP構成の具体方針

### 10.1 MVPで採用する構成

- Webアプリ: 1系統
- APIサービス: 1サービス
- Recoサービス: 1サービス
- PostgreSQL: Neon
- Redis: 単一
- Batch: 定期ジョブ
- Embedding API: 外部利用
- 監視: 基礎監視 + アプリメトリクス

### 10.2 MVPでやりすぎないこと

- Reco内部の過剰なマイクロサービス分割
- DB読み取り分離
- Redis Cluster
- 高度な分散トレーシング
- 複雑な自動スケーリング

---

## 11. 将来拡張方針

### 11.1 拡張ステップ

| フェーズ | 拡張内容                        |
| -------- | ------------------------------- |
| Phase1   | MVP最小構成                     |
| Phase2   | API / Reco 複数インスタンス化   |
| Phase3   | DB Replica、Redis Cluster       |
| Phase4   | 分析ストア分離、CDN / Edge 強化 |

### 11.2 拡張トリガー

- API p95 latency 悪化
- Reco CPU使用率増加
- DB読み取り負荷増加
- 監視・集約クエリの競合増加

## 13. 設計原則まとめ

```
・API / Reco / Batch を責務単位で分離する
・正本は PostgreSQL に集約する
・低レイテンシ参照は Redis で補助する
・MVPでは最小構成に留める
・将来の水平分散に耐えられる配置にする
・監視は基盤だけでなく品質まで対象とする
```

本サービスの基盤構成は、
「Web / API / Reco / PostgreSQL / Redis / Batch」を最小単位として分離し、

将来の性能・品質・改善要求に応じて段階的に拡張できる構成とする

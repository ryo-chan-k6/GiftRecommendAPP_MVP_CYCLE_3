## 1. 目的

本ドキュメントは、本サービスにおけるキャッシュの対象・粒度・TTL・更新方式・失効（invalidate）方法を定義し、**性能要件（低レイテンシ）とデータ鮮度のバランス**を最適化することを目的とする。

---

## 2. 適用範囲

| レイヤ | 対象                           |
| ------ | ------------------------------ |
| Reco   | embedding / retrieval / config |
| API    | レスポンス（将来）             |
| DB     | クエリ結果（間接的に）         |

---

## 3. 設計原則

```
・重い処理のみキャッシュする（無差別キャッシュ禁止）
・TTL + イベント駆動の併用
・キーは「再現可能」にする
・キャッシュは破棄されても正しく動く（必須）
・キャッシュヒット率を観測する
```

---

## 4. キャッシュ対象一覧（確定版）

| 対象            | 優先度     | キャッシュ種別   | 保存場所       |
| --------------- | ---------- | ---------------- | -------------- |
| item_embedding  | 高         | 永続系キャッシュ | メモリ / Redis |
| user_embedding  | 高         | 短期キャッシュ   | Redis          |
| retrieval結果   | 高         | 短期キャッシュ   | Redis          |
| semantic_config | 高         | 常駐キャッシュ   | プロセス内     |
| model_version   | 高         | 常駐キャッシュ   | プロセス内     |
| レコメンド結果  | 低（将来） | 短期キャッシュ   | Redis          |

---

## 5. TTL・更新ポリシー（詳細）

### 5.1 商品embedding

| 項目       | 内容                 |
| ---------- | -------------------- |
| TTL        | 24時間               |
| 更新       | BTで再生成時に上書き |
| invalidate | embedding更新時      |
| キー       | `embedding:{itemId}` |
| 保存       | Redis or メモリ      |

### 設計意図

- 再計算コストが高い
- 鮮度要求は低い

---

### 5.2 Retrieval結果

| 項目       | 内容                          |
| ---------- | ----------------------------- |
| TTL        | 5分                           |
| 更新       | TTL満了 or ミス時             |
| invalidate | 不要（自然失効）              |
| キー       | `retrieval:{hash(condition)}` |
| 保存       | Redis                         |

### 設計意図

- 同一条件の連続検索を高速化
- 鮮度とのバランス

---

### 5.3 semantic_config

| 項目       | 内容                 |
| ---------- | -------------------- |
| TTL        | 無期限               |
| 更新       | 管理APIで更新時      |
| invalidate | config変更時         |
| 保存       | メモリ（プロセス内） |

### 設計意図

- 読み取り頻度高い
- 更新は明示的

---

### 5.4 model_version

| 項目       | 内容          |
| ---------- | ------------- |
| TTL        | 無期限        |
| 更新       | version切替時 |
| invalidate | 切替時        |
| 保存       | メモリ        |

---

### 5.5 レコメンド結果（将来）

| 項目 | 内容                               |
| ---- | ---------------------------------- |
| TTL  | 1〜5分                             |
| キー | `recommendation:{hash(condition)}` |
| 利用 | 高頻度クエリ                       |

## 6. キャッシュキー設計

### 6.1 原則

```
・同一入力 → 同一キー
・順序や表記ゆれを吸収する
・ハッシュ化する
```

---

### 6.2 retrievalキー生成

```
{
  "relationship":"boss",
  "occasion":"birthday",
  "budgetRange":"3000-5000",
  "preferred": ["高級感","実用性"],
  "ng": ["安っぽい"]
}
```

👇

```
hash(JSON.stringify(sorted_condition))
```

---

### 6.3 注意点

- 配列順序はソート
- 空項目は除外
- 正規化必須

---

## 7. invalidate戦略

### 7.1 種類

| 種類         | 対象           | 方法     |
| ------------ | -------------- | -------- |
| TTL失効      | retrieval      | 自然失効 |
| イベント更新 | config         | 即時更新 |
| バッチ更新   | item_embedding | 上書き   |
| 入力条件更新 | user_embedding | 上書き   |

---

### 7.2 設計原則

```
・TTLだけに依存しない
・重要データはイベントで更新
・invalidateは最小限にする
```

---

## 8. フォールバック

```
・キャッシュミス時は通常処理
・キャッシュ障害時は無視して継続
```

👉 キャッシュは「補助」であり「必須依存ではない」

## 11. リスクと対策

| リスク     | 対策             |
| ---------- | ---------------- |
| 古いデータ | TTL + invalidate |
| メモリ枯渇 | LRU              |
| キー爆発   | 条件制限         |
| 不整合     | 再計算可能設計   |

```
・重い処理だけキャッシュ

```

```
「依存しない設計」を前提に、
TTLとイベント更新で制御する
```

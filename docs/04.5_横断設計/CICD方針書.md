## 1. 目的

本方針書は、Gift Recommendation Service の開発・テスト・デプロイを自動化し、**品質と開発速度を両立するCI/CD方針**を定義するものである。

本方針書の目的は以下である。

- 品質ゲートを自動化する
- リリースの安全性を担保する
- 手動作業を最小化する
- テスト方針・ブランチ戦略と整合させる
- MVP開発に適した軽量CI/CDを構築する

---

## 2. 前提

| 項目          | 内容                          |
| ------------- | ----------------------------- |
| 開発体制      | 個人開発                      |
| Repo          | monorepo                      |
| ブランチ戦略  | main / develop / feature      |
| テスト方針    | 重点集中型                    |
| Observability | MVPから重要                   |
| docs          | 正本                          |
| デプロイ      | Web / API / Reco / Batch 分離 |

---

## 3. CI/CD基本方針

---

### 3.1 方針

- CIは「品質ゲート」として機能させる
- CDは「安全にリリースするための制御」とする
- 自動化は最小構成から始める
- 手動判断を完全には排除しない
- Observability成立をデプロイ条件に含める

---

### 3.2 事実と推論

### ■ 事実

- テストは完全網羅しない（MVP）
- 個人開発である
- レコメンド品質はObservabilityに依存

### ■ 推論

👉

**CI/CDは軽量だが「壊れない保証」と「観測可能性」は必須**

# 5. CI（継続的インテグレーション）

---

## 5.1 目的

- コード品質の担保
- 不具合の早期検知
- マージ可否判断

---

## 5.2 実行タイミング

| タイミング    | 実行 |
| ------------- | ---- |
| PR作成        | 実行 |
| PR更新        | 実行 |
| developマージ | 実行 |
| mainマージ    | 実行 |

---

## 5.3 CI実行内容

---

### ① build

- 全アプリのビルド
- 型チェック

---

### ② lint

- コーディング規約チェック
- フォーマットチェック

---

### ③ Unitテスト

- スコア計算
- ロジック確認

---

### ④ Integrationテスト

- API
- DB
- Reco連携

---

### ⑤ 基盤チェック

- 環境変数検証
- DB接続
- API疎通

---

### ⑥ Observability基本チェック（重要）

- runId生成確認
- ログ出力確認
- メトリクス出力確認

---

## 5.4 CI成功条件

以下すべて満たすこと：

- build成功
- lint成功
- Unit成功
- Integration成功
- 基盤チェック成功

---

# 6. CD（継続的デリバリー / デプロイ）

---

## 6.1 方針

- 完全自動デプロイは行わない（MVP）
- **半自動（CI + 手動承認）方式を採用**
- 本番反映は明示的に行う

---

## 6.2 デプロイ対象

| コンポーネント | デプロイ             |
| -------------- | -------------------- |
| Web            | 自動 or 半自動       |
| API            | 半自動               |
| Reco           | 半自動               |
| Batch          | 手動 or スケジュール |

---

## 6.3 環境構成

| 環境        | 用途     |
| ----------- | -------- |
| local       | 開発     |
| develop相当 | 統合確認 |
| production  | 本番     |

---

## 6.4 デプロイフロー

---

### develop → 確認

- 統合確認
- 手動確認

---

### main → 本番

```

```

---

## 6.5 デプロイ条件

以下をすべて満たす：

- CI成功
- テスト完了
- 主要導線確認
- docs更新済み
- Observability成立

---

# 7. Observability連携（重要）

---

## 7.1 位置づけ

👉 ObservabilityはCDの「品質ゲートの一部」

---

## 7.2 デプロイ後確認

必須確認項目：

- runIdトレース可能
- APIログ出力
- メトリクス出力
- 分布取得可能
- ダッシュボード反映

---

## 7.3 NG条件

以下の場合はリリース不可：

- メトリクス未出力
- runId欠落
- 分布取得不可
- ログ欠損

---

# 8. ブランチ戦略との連携

---

## 8.1 feature / fix

- CIのみ実行
- CDは行わない

---

## 8.2 develop

- CI実行
- 統合確認用デプロイ（必要に応じて）

---

## 8.3 main

- CI実行
- 本番デプロイ対象

---

## 8.4 hotfix

- main起点
- 最優先CI
- 即デプロイ

---

# 9. docs正本との連携

---

## 9.1 方針

- docs更新をCIでチェック対象にする

---

## 9.2 チェック例

- API変更 → docs更新されているか
- 設計変更 → docs差分あるか

---

# 10. 自動化範囲

---

## 10.1 自動化する

| 項目              | 内容 |
| ----------------- | ---- |
| build             | 必須 |
| lint              | 必須 |
| Unit              | 必須 |
| Integration       | 必須 |
| 一部Observability | 必須 |

---

## 10.2 自動化しない

| 項目             | 理由      |
| ---------------- | --------- |
| UI確認           | 手動      |
| レコメンド妥当性 | 人間判断  |
| 分布評価         | 手動      |
| 負荷試験         | MVP対象外 |

---

# 11. 失敗時の対応

---

## CI失敗

- 修正必須
- マージ禁止

---

## デプロイ失敗

- ロールバック
- Issue化

---

## Observability不備

- リリース停止
- 修正対応

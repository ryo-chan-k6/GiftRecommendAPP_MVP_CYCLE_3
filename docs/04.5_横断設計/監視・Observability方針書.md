## 1. 目的

本ドキュメントは、本サービスにおける監視 / Observability の基本方針を定義し、

システムの安定稼働、障害対応、性能維持、レコメンド品質維持、および継続的改善を可能にするための監視思想を明確化することを目的とする。

本方針書は、単なるインフラ監視ではなく、**レコメンド品質監視を含む全体観測方針**を対象とする。

---

## 2. 監視 / Observability の位置づけ

本サービスにおける Observability は、以下の3目的を持つ。

| 目的     | 内容                                                       |
| -------- | ---------------------------------------------------------- |
| 障害検知 | システム停止・性能劣化・異常動作を早期に検知する           |
| 原因特定 | どこで、なぜ問題が起きたかを追跡可能にする                 |
| 品質改善 | レコメンド品質の劣化や偏りを継続的に把握し、改善に接続する |

---

## 3. 基本思想

### 3.1 基本方針

```
・監視は「システムが生きているか」だけでは不十分とする
・監視は「サービスが価値を提供できているか」まで対象とする
・監視は「検知」で終わらず、「原因特定」と「改善」に接続する
・基盤監視・アプリ監視・業務監視・モデル監視を分けて考える
・最重要の追跡単位は runId とする
```

### 3.2 本サービス特有の思想

```
本サービスのObservabilityは、
インフラ監視ではなく「意味空間の健全性監視」まで含む
```

理由：

- レコメンドの価値が、単なるAPI応答ではなく意味空間・特徴量空間に依存するため
- APIが200を返していても、推薦品質が崩壊している可能性があるため
- 分布異常、空間異常、スコアバランス異常を監視対象に含める必要があるため

---

## 4. 監視レイヤ方針

監視対象は、以下の4レイヤで整理する。

| レイヤ       | 監視対象                         | 主目的                   |
| ------------ | -------------------------------- | ------------------------ |
| システム監視 | uptime, CPU, memory, restart     | 稼働確認                 |
| アプリ監視   | latency, error, throughput       | 障害・性能劣化検知       |
| 業務監視     | success_rate, fallback_rate, CTR | サービス品質確認         |
| モデル監視   | 分布、空間、スコア構造           | レコメンド品質の異常検知 |

### 4.1 システム監視

- プロセス生存
- コンテナ再起動
- CPU / メモリ使用率
- ネットワークエラー
- 基盤ヘルスチェック

### 4.2 アプリ監視

- API応答時間
- APIエラー率
- Reco処理時間
- Reco失敗率
- DBクエリ遅延
- キャッシュヒット率 / ミス率

### 4.3 業務監視

- recommendation_success_rate
- fallback_rate
- empty_result_rate
- CTR
- favorite_rate
- human_eval_completion_rate

### 4.4 モデル監視

- item_feature分布
- user_feature分布
- social / symbolic 空間バランス
- meaning_diff分布
- context / popularity / risk の寄与バランス
- 正規化統計量（μ / σ）
- user × item 相互作用の異常

---

## 5. 観測の3本柱

Observability は、以下の3手段を組み合わせて実現する。

| 手段       | 役割                |
| ---------- | ------------------- |
| ログ       | 個別事象の記録      |
| メトリクス | 継続監視・閾値管理  |
| トレース   | 1件の処理の経路追跡 |

### 5.1 ログの位置づけ

- 何が起きたかを残す
- 障害時の詳細調査に使う
- 管理操作や監査にも使う

### 5.2 メトリクスの位置づけ

- 監視・ダッシュボード・アラートに使う
- 傾向変化の把握に使う
- システム / 業務 / モデルを横断して観測する

### 5.3 トレースの位置づけ

- requestId / runId 単位で経路を追えるようにする
- API → Reco → DB / Cache の経路を追跡可能にする
- 失敗地点と最後の成功ステップを判別可能にする

---

## 6. 追跡キー方針

### 6.1 基本方針

```
・リクエスト単位は requestId
・推薦処理単位は runId
・評価単位は taskId / evalResultId
・バッチ単位は batchRunId
```

### 6.2 最重要キー

**runId** を中心に、以下を横断接続できるようにする。

- recommendation_run
- recommendation_result
- run_event_log
- interaction_log
- metric_log
- エラー記録
- 品質監視集計

---

## 7. 監視対象選定方針

### 7.1 監視対象の選定基準

以下のいずれかを満たすものを監視対象とする。

| 基準              | 内容                       |
| ----------------- | -------------------------- |
| 障害影響が大きい  | 停止や性能劣化に直結する   |
| 劣化が見えにくい  | API 200 でも品質が落ちうる |
| 改善ループに使う  | 調整・評価・比較に必要     |
| 高頻度 / 高コスト | コストや負荷に影響する     |

### 7.2 監視対象に含めるべきもの

- APIレイテンシ
- Recoレイテンシ
- fallback_rate
- feature分布
- μ / σ
- score構成
- interaction指標

### 7.3 監視対象に含めすぎないもの

- 使わない詳細ログ
- 改善や検知に使わないメトリクス
- 粒度が細かすぎて運用できない監視項目

---

## 8. アラート方針

### 8.1 基本方針

```
・アラートは「即時対応が必要なもの」に絞る
・情報収集用メトリクスと即時通知対象を分ける
・品質監視は即時アラートと定期レビューを使い分ける
```

### 8.2 アラート分類

| 種類         | 例                    | 対応               |
| ------------ | --------------------- | ------------------ |
| 障害アラート | API停止、Reco失敗急増 | 即時通知           |
| 性能アラート | p95 latency超過       | 即時または優先通知 |
| 品質アラート | fallback_rate急増     | 優先通知           |
| 傾向監視     | CTR低下、分布偏り     | 定期レビュー       |

### 8.3 アラートの考え方

- **障害**は即時通知
- **性能**は閾値超過で通知
- **品質**は即時通知と日次レビューを併用
- **モデル監視**は急激な異常は通知、緩やかな劣化は定期レビュー

---

## 9. ダッシュボード方針

### 9.1 分離方針

ダッシュボードは以下に分ける。

| ダッシュボード       | 主利用者           | 内容                                      |
| -------------------- | ------------------ | ----------------------------------------- |
| 技術ダッシュボード   | 開発者 / 運用者    | CPU, memory, latency, error               |
| 業務ダッシュボード   | PM / 分析者        | success_rate, CTR, favorite_rate          |
| モデルダッシュボード | レコメンド改善担当 | feature分布, μ/σ, 空間構造, scoreバランス |

### 9.2 基本方針

```
・1画面で全て見せない
・役割ごとに必要な指標を整理する
・「今壊れているか」と「なぜ壊れているか」を分けて見せる
```

---

## 10. 品質監視方針（本サービス固有）

### 10.1 基本方針

```
・レコメンド品質監視をObservabilityの中心に置く
・分布は「正解」ではなく「異常検知センサー」として扱う
・正規化統計量は座標系として管理する
・品質監視は必ずFB分類と改善に接続する
```

### 10.2 監視対象

- item_feature分布
- user_feature分布
- social_mean / symbolic_mean
- meaning_diff
- context_score
- final_score
- score構成比率
- FB分類結果
- human_eval結果

### 10.3 接続先

品質監視で異常を検知したら、以下に接続する。

```
観測
↓
異常検知
↓
FB分類
↓
修正対象特定
↓
改善
```

## 11. 基盤観測とアプリ観測の責務分担

### 11.1 基盤側に任せるもの

- uptime
- CPU / memory
- restart_count
- network_error
- プロセス生存
- HTTPレベルの基本指標

### 11.2 アプリ側で明示設計するもの

- recommendation_success_rate
- fallback_rate
- empty_result_rate
- Reco段階別latency
- feature分布
- scoreバランス
- FB分類結果

### 11.3 方針

```
・基盤メトリクスだけでは業務品質は監視できない
・業務・モデル観測はアプリ側で明示的に設計する
```

---

## 12. ログ・メトリクス設計への引継ぎ方針

### 12.1 ログ設計へ引き継ぐもの

- run_event_log の粒度
- interaction_log の記録方針
- metric_log の記録項目
- error_log / audit_log の対象

### 12.2 メトリクス設計へ引き継ぐもの

- 基盤メトリクス一覧
- アプリメトリクス一覧
- 業務メトリクス一覧
- モデルメトリクス一覧

### 12.3 重要な引継ぎ事項

```
・非機能要件で定義した指標は、ログ設計書で具体項目化する
・ダッシュボード対象は集約テーブルまたは監視基盤への出力前提で設計する
```

## 13. MVP方針

### 13.1 MVPで必須

- requestId / runId 追跡
- API / Reco latency監視
- error_rate監視
- fallback_rate監視
- item / user feature 分布の簡易監視
- μ / σ の保持と比較
- interaction 指標の基本集計

### 13.2 MVPでは後回し可

- 高度な分散トレーシング
- 自動異常検知
- 高度な相互作用可視化
- FB分類の完全自動化

```

・Observabilityは改善ループまで含めて設計する
```

## 1. 目的

本資料は、Gift Recommendation Service における API の設計方針を定義することを目的とする。

本資料で扱う対象は、**外部公開APIおよびサブシステム間APIの設計原則**であり、個別エンドポイントの Request / Response 詳細、JSON Schema、認証設定値、エラーコード詳細は後続の API仕様書で定義する。

本資料は、インターフェース一覧で整理した **Web / Auth / API / Reco / Admin-Eval / Batch / DB / Log** の境界のうち、特に **APIとして公開・利用する境界** の設計原則を定めるための成果物である。

---

## 2. スコープ

### 2-1. 対象

本方針書の対象は以下とする。

| 対象        | 内容                                         |
| ----------- | -------------------------------------------- |
| Web向けAPI  | 利用者UIから呼び出される業務API              |
| 管理向けAPI | 管理者 / 評価者が利用する設定・評価系API     |
| 内部連携API | API → Reco など、サブシステム間で呼び出すAPI |
| ログ起点API | interaction / run_event などの記録系API      |

### 2-2. 非対象

本方針書では以下は扱わない。

| 非対象                                    | 後続成果物                |
| ----------------------------------------- | ------------------------- |
| Reco内部モジュール間の入出力              | モジュール別入出力定義表  |
| ETL / 特徴量 / embedding / 評価ジョブ詳細 | バッチ仕様書              |
| ログテーブルの詳細定義                    | ログ設計書                |
| 物理テーブル定義 / DDL                    | テーブル一覧、物理ER、DDL |

## 3. API設計の基本方針

### 3-1. 方針の全体像

API設計の基本原則は以下とする。

| 方針             | 内容                                                                                |
| ---------------- | ----------------------------------------------------------------------------------- |
| 役割分離         | APIは「ユーザー要求に対する意思決定と応答」を担い、材料生成・集計・改善はBTに委ねる |
| 境界重視         | APIはサブシステム境界に立つ契約であり、内部アルゴリズムを直接露出しない             |
| 低レイテンシ優先 | APIは軽量・同期的に完了する範囲に責務を限定する                                     |
| 契約安定性重視   | クライアント都合で頻繁に壊れないAPI契約を優先する                                   |
| 再現性確保       | recommendation_run / run_event / interaction の記録と参照可能性を担保する           |
| 将来拡張性       | MVP段階では最小公開、将来の評価・管理・実験機能追加に耐える命名と構造にする         |

### 3-2. 事実

インターフェース一覧では、IFは「内部アルゴリズム単位」ではなく「サブシステム境界 / 永続化境界 / 手動運用境界」で整理する方針です。

### 3-3. 推論

したがって API でも、`Feature Distance計算` や `MMR多様化` のような内部処理は個別APIとして公開せず、**業務上意味のある単位**で公開するのが適切である。

## 4. APIの責務範囲

### 4-1. APIが担う責務

| 責務                     | 内容                                                |
| ------------------------ | --------------------------------------------------- |
| リクエスト受領           | 入力受領、基本バリデーション、認証・認可確認        |
| 業務オーケストレーション | 必要な内部処理の呼び出し順制御                      |
| 同期応答生成             | UIが必要とする応答データ整形                        |
| 即時記録                 | recommendation_run、run_event、interaction 等の記録 |
| 軽量判定                 | A/B割当、設定読込、軽量ルール判定                   |
| 契約保証                 | Request / Response / Error の安定した外部契約提供   |

### 4-2. APIが担わない責務

| 非責務                             | 委譲先                    |
| ---------------------------------- | ------------------------- |
| 商品収集・ETL                      | バッチ                    |
| 商品特徴量生成                     | バッチ                    |
| embedding生成                      | バッチ                    |
| 評価データセット生成               | バッチ                    |
| 指標集計                           | バッチ                    |
| 高コストな再計算                   | バッチ                    |
| Reco内部アルゴリズムの中間処理公開 | モジュール設計 / 内部実装 |

### 4-3. 設計判断

OL/BT責務整理表の原則どおり、APIはユーザーリクエスト起点の処理を担い、重い処理は含めない。重い処理が必要な場合は、**事前計算結果・キャッシュ・BT生成済みデータ**を利用する。

## 5. API分類方針

本サービスのAPIは、用途別に以下へ分類する。

| API分類     | 主用途                                           | 想定利用者      |
| ----------- | ------------------------------------------------ | --------------- |
| 推薦API     | レコメンド実行、結果取得                         | Webクライアント |
| 行動記録API | impression / click / favorite 等の記録           | Webクライアント |
| 実験API     | A/B割当、実験条件取得                            | Web / API内部   |
| 管理API     | semantic_config / model_version などの参照・更新 | 管理者          |
| 評価API     | human_eval_task参照、human_eval結果登録          | 評価者 / 管理者 |
| 履歴参照API | recommendation_run参照、実行履歴確認             | 管理者 / 運用者 |
| 内部連携API | API → Reco などのサブシステム間連携              | システム内部    |

## 6. API公開粒度の方針

### 6-1. 公開単位

APIは**業務ユースケース単位**で公開する。

内部アルゴリズム単位では公開しない。

### 6-2. 例

| 公開するAPI               | 公開しないAPI                 |
| ------------------------- | ----------------------------- |
| `/recommendations`        | `/feature-distance/calculate` |
| `/interactions`           | `/social-symbolic/aggregate`  |
| `/admin/semantic-configs` | `/risk-score/calculate`       |
| `/human-evals`            | `/mmr/diversify`              |

### 6-3. 設計理由

インターフェース一覧で、内部モジュール計算単位はIF一覧から除外し、後続のモジュール別入出力定義表等へ委ねる方針が採られています。APIも同じ原則に従うべきです。

---

## 7. URL設計方針

### 7-1. 基本方針

| 方針         | 内容                                           |
| ------------ | ---------------------------------------------- |
| 名詞中心     | URLは原則として名詞で表現する                  |
| 複数形基本   | コレクションを表す場合は複数形を基本とする     |
| 階層最小化   | ネストは必要最小限にとどめる                   |
| 業務単位優先 | 技術処理名ではなく業務対象名を優先する         |
| 管理系分離   | 管理APIは `/admin/...` に集約する              |
| 評価系分離   | 評価運用APIは `/eval/...` などで分離可能にする |

### 7-2. 例

| 用途         | 例                             |
| ------------ | ------------------------------ |
| 推薦実行     | `/recommendations`             |
| 行動記録     | `/interactions`                |
| 実験割当     | `/experiments/assignments`     |
| 設定管理     | `/admin/semantic-configs`      |
| モデル管理   | `/admin/model-versions`        |
| human_eval   | `/eval/tasks`, `/eval/results` |
| 実行履歴参照 | `/admin/recommendation-runs`   |

## 8. Request / Response 設計方針

### 8-1. Request方針

| 方針         | 内容                                       |
| ------------ | ------------------------------------------ |
| 明示的入力   | クライアントが指定すべき値は明示的に受ける |
| 内部推定隠蔽 | 内部推定パラメータや内部重みは原則非公開   |
| 将来拡張余地 | 任意条件は拡張しやすい構造にする           |
| 冪等性配慮   | 記録系APIは必要に応じて冪等キーを考慮する  |

### 8-2. Response方針

| 方針           | 内容                                     |
| -------------- | ---------------------------------------- |
| UI最適化       | UIでそのまま使いやすい構造を優先する     |
| 内部詳細非露出 | 内部アルゴリズムの中間値は原則返さない   |
| 追跡可能性     | 必要に応じて `run_id` 等の追跡キーを返す |
| バージョン耐性 | 将来フィールド追加しやすい応答構造にする |

### 8-3. 中間値公開方針

MVP段階では、内部スコアのすべてを公開しない。

ただし、管理APIや評価APIでは、検証目的で限定的に返却する余地を持つ。

## 9. 認証・認可方針

### 9-1. API区分別方針

| API分類     | 認証                               | 認可             |
| ----------- | ---------------------------------- | ---------------- |
| 推薦API     | 利用者認証前提またはセッション識別 | 一般利用者権限   |
| 行動記録API | 利用者識別前提                     | 自身の操作のみ   |
| 管理API     | 管理者認証必須                     | 管理者のみ       |
| 評価API     | 評価者または管理者認証必須         | 評価運用権限のみ |
| 内部連携API | サービス間認証                     | 内部サービスのみ |

### 9-2. 方針

Web ↔ Auth / API の境界はインターフェース一覧の主要対象であるため、API設計では **外部向け認証** と **内部サービス間認証** を明確に分ける。

---

## 10. エラー設計方針

### 10-1. 基本方針

| 方針                    | 内容                                           |
| ----------------------- | ---------------------------------------------- |
| HTTP準拠                | 通信失敗と業務失敗をHTTPステータスで大別する   |
| 業務エラー明確化        | 入力不正・権限不足・未取得・内部失敗を区別する |
| 利用者向け/運用向け分離 | 利用者向けメッセージと内部原因詳細を分離する   |
| run_id連携              | 内部調査用に run_id / request_id を持たせる    |
| 再試行可否意識          | クライアント再試行可否を判別しやすくする       |

### 10-2. エラー分類例

| 分類       | 例                         |
| ---------- | -------------------------- |
| 入力エラー | 必須項目不足、形式不正     |
| 認証エラー | 未認証、期限切れ           |
| 認可エラー | 権限不足                   |
| 業務エラー | 対象なし、無効状態         |
| 内部エラー | 推薦処理失敗、内部連携失敗 |

## 11. ログ・監査連携方針

### 11-1. 基本方針

APIは、後続の分析・監視・再現性を担保するため、必要な記録を残す。

### 11-2. 記録対象

| 記録対象           | 内容                                   |
| ------------------ | -------------------------------------- |
| recommendation_run | レコメンド要求と結果のスナップショット |
| run_event_log      | API実行上のステップイベント            |
| interaction_log    | ユーザー行動イベント                   |
| metric_log         | レイテンシ、エラー件数等のメトリクス   |

### 11-3. 方針

ログ詳細はログ設計書で定義するが、API設計としては「どのAPIがどの記録を発火するか」を明確にする。

また、Hybrid処理は **APIでイベントを発火し、BTがそれを後続処理トリガーとして受ける** 前提で設計する。

---

## 12. Reco連携方針

### 12-1. 基本方針

APIは Reco の内部アルゴリズムを直接外部公開しない。

API は **業務オーケストレータ** として振る舞い、Reco には必要最小限の入力を渡し、結果を業務API応答へ整形して返却する。

### 12-2. 連携原則

| 項目       | 方針                                   |
| ---------- | -------------------------------------- |
| API → Reco | 内部連携専用                           |
| 外部公開   | しない                                 |
| 入力       | APIで整形済みの条件                    |
| 出力       | 候補商品、スコア、必要最小限の説明情報 |
| 障害時     | APIがエラー分類して利用者応答へ変換    |

---

## 13. バージョニング方針

### 13-1. 基本方針

外部API契約は将来変更を見越してバージョニング可能な構造とする。

### 13-2. MVP方針

MVPでは大きな破壊的変更を想定しない場合でも、`/api/v1/...` のように **経路上のバージョン** を持たせる方針を推奨する。

### 13-3. 理由

推論として、推薦ロジックや評価運用は今後変更可能性が高く、外部契約の安定性を確保する必要があるため。

---

## 14. 非機能方針

| 観点   | 方針                                   |
| ------ | -------------------------------------- |
| 性能   | 推薦APIは低レイテンシを優先            |
| 可用性 | API失敗時も原因追跡可能であること      |
| 監視性 | request_id / run_id ベースで追跡可能   |
| 保守性 | API公開契約と内部アルゴリズムを分離    |
| 拡張性 | 管理・評価・実験機能の追加に耐える構造 |
| 安全性 | 権限境界を明確化し、管理系APIを分離    |

## 16. 一言まとめ

本サービスのAPIは、**内部アルゴリズムを公開するためのものではなく、ユーザー要求に対して安定した業務契約を提供するための境界**として設計する。

したがって、APIは **軽量・同期・業務単位** に保ち、**重い処理・集計・改善はBTへ委譲**し、**記録と追跡可能性はAPIで担保**する。

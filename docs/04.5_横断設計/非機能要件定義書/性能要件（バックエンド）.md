## 1. 目的

本ドキュメントは、本サービスにおける性能要件を定義し、

API設計・Reco設計・基盤設計・キャッシュ設計の前提条件を明確化することを目的とする。

---

## 2. 対象範囲

| 対象 | 内容                      |
| ---- | ------------------------- |
| API  | 外部公開API               |
| Reco | レコメンド処理（内部API） |
| DB   | データ取得・保存          |
| Log  | ログ記録                  |

---

## 3. 性能要件サマリ（SLO）

### 3.1 レスポンス時間

| API           | p50   | p95     | p99     | 最大    |
| ------------- | ----- | ------- | ------- | ------- |
| レコメンドAPI | 800ms | 2,000ms | 4,000ms | 5,000ms |

---

### 3.2 スループット

| 指標           | 値               |
| -------------- | ---------------- |
| 同時ユーザー数 | 50〜100（MVP）   |
| RPS            | 10〜30           |
| 将来拡張       | 水平スケール前提 |

---

## 4. レコメンド処理時間内訳（重要）

レコメンドAPIのレスポンスは以下の分解で設計する。

```
API受信
↓
Reco処理
  ├ 入力解析
  ├ User Meaning推定
  ├ Candidate Retrieval
  ├ Ranking
↓
レスポンス生成
```

---

### 4.1 各ステップの目標時間

| ステップ       | 目標時間 |
| -------------- | -------- |
| 入力解析       | 50ms     |
| Meaning推定    | 300ms    |
| Retrieval      | 300ms    |
| Ranking        | 300ms    |
| レスポンス整形 | 50ms     |

👉 合計目標：1,000ms前後（通常）

## 7. キャッシュ戦略

### 7.1 キャッシュ対象

| 対象            | 方針           |
| --------------- | -------------- |
| 商品embedding   | キャッシュ     |
| retrieval結果   | 短期キャッシュ |
| semantic_config | 常駐           |
| model_version   | 常駐           |

---

### 7.2 キャッシュTTL

| 対象            | TTL                        | 更新トリガー                    | 設計意図                                     |
| --------------- | -------------------------- | ------------------------------- | -------------------------------------------- |
| 商品embedding   | **24時間（86400秒）**      | BTでembedding再生成時に強制更新 | 商品情報は頻繁に変わらないため長期キャッシュ |
| retrieval結果   | **5分（300秒）**           | TTL満了 or キャッシュミス       | 同一条件の短時間再検索を高速化               |
| semantic_config | **無期限（プロセス常駐）** | config更新時に即時リロード      | 実行時の安定性・高速化                       |
| model_version   | **無期限（プロセス常駐）** | version切替時に即時反映         | 推論ロジックの一貫性確保                     |

## 9. ボトルネック対策

### 9.1 想定ボトルネック

| 対象          | 内容        |
| ------------- | ----------- |
| embedding計算 | 重い        |
| Retrieval     | DBアクセス  |
| Ranking       | O(n)〜O(n²) |

---

### 9.2 対策

| 対策         | 内容       |
| ------------ | ---------- |
| 事前計算     | 商品特徴量 |
| キャッシュ   | embedding  |
| 件数制限     | top-k制限  |
| インデックス | DB最適化   |

---

## 10. フォールバック設計

### 10.1 フォールバック条件

| 条件          | 対応              |
| ------------- | ----------------- |
| Retrieval失敗 | 人気商品返却      |
| Ranking失敗   | Retrieval順で返却 |

---

### 10.2 フォールバック要件

```
・必ず何かしらの推薦結果を返す
・空レスポンスは禁止
```

---

## 11. リトライ設計

### 11.1 対象

| 対象        | 条件          |
| ----------- | ------------- |
| Reco内部API | timeout / 5xx |
| DBアクセス  | 一時エラー    |

---

### 11.2 リトライ仕様

| 項目 | 内容                |
| ---- | ------------------- |
| 回数 | 最大2回             |
| 間隔 | 100ms → 300ms       |
| 方式 | exponential backoff |

---

## 12. 非同期化ポリシー

```
・レコメンド処理は同期（UX優先）
・重い前処理はBTへオフロード
```

---

### 13.1 必須メトリクス

| ------------ | ------------ |
| latency      | API応答時間  |
| error_rate   | エラー率     |
| throughput   | RPS          |
| reco_latency | Reco処理時間 |

---

## 14. ログ連携要件

| ログ          | 内容           |
| ------------- | -------------- |
| run_event_log | 各ステップ時間 |
| metric_log    | latency記録    |
| error_log     | エラー内容     |

---

## 15. 設計原則（性能）

```
・同期処理は2秒以内を目標
・重い処理はBTへ逃がす
・キャッシュを積極利用
・Reproducibilityを優先
・フォールバック必須
```

---

# 16. 一言まとめ

```
レコメンドは同期で高速応答しつつ、
重い処理は事前計算・キャッシュ・並列化で吸収し、
最悪でもフォールバックで結果を返す設計とする
```

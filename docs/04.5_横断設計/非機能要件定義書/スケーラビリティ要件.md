## 1. 目的

本ドキュメントは、本サービスの負荷増加に対する拡張性（スケーラビリティ）を定義し、

性能劣化を防ぎつつ段階的にシステムを拡張可能とする設計指針を示すことを目的とする。

---

## 2. スコープ

| 対象  | 内容               |
| ----- | ------------------ |
| API   | 外部リクエスト処理 |
| Reco  | 推薦処理           |
| DB    | データストア       |
| Cache | Redis / メモリ     |
| Batch | データ生成         |

---

## 3. スケーラビリティ設計原則

```
・水平スケールを前提とする
・ステートレス設計を徹底する
・ボトルネックを分離する
・段階的スケール（過剰設計しない）
・重い処理は事前計算へ逃がす
```

---

### 4.2 将来想定

| -------------- | -------- |
| 同時ユーザー数 | 1,000+   |

---

## 5. スケール対象と方針

---

### 5.1 コンポーネント別

| コンポーネント | スケール方式 | 方針           |
| -------------- | ------------ | -------------- |
| API            | 水平         | ステートレス化 |
| Reco           | 水平         | CPUスケール    |
| DB             | 垂直 → 水平  | 初期は単一     |
| Cache          | 水平         | Redis          |
| Batch          | 水平         | 並列処理       |

---

## 6. APIスケーラビリティ

---

### 6.1 要件

| 項目         | 内容     |
| ------------ | -------- |
| ステートレス | 必須     |
| セッション   | 持たない |
| スケール方式 | 水平     |
| LB           | 将来導入 |

---

### 6.2 スケールトリガー

| 条件        | 対応             |
| ----------- | ---------------- |
| CPU > 70%   | インスタンス追加 |
| latency悪化 | スケールアウト   |

---

## 7. Recoスケーラビリティ（重要）

---

### 7.1 要件

| 項目         | 内容         |
| ------------ | ------------ |
| 分離         | APIと独立    |
| スケール     | 水平         |
| ボトルネック | CPU / メモリ |

---

### 7.2 スケール戦略

```
単一インスタンス
↓
複数インスタンス
↓
ロード分散
```

---

### 7.3 注意点

```
・内部モジュールは分離しない（MVP）
・サービス単位でスケール
```

---

## 11. ボトルネック分析

---

### 11.1 想定

| 項目      | 内容    |
| --------- | ------- |
| Reco      | CPU負荷 |
| Retrieval | DB I/O  |
| Ranking   | 計算量  |
| Cache     | メモリ  |

---

### 11.2 対策

| 対策         | 内容       |
| ------------ | ---------- |
| キャッシュ   | 高優先     |
| 件数制限     | top-k      |
| 非同期化     | BTへ逃がす |
| インデックス | DB最適化   |

---

## 12. スケールトリガー定義

---

### 12.1 指標

| 指標     | 閾値      |
| -------- | --------- |
| CPU      | 70%以上   |
| latency  | p95 > 2秒 |
| エラー率 | > 2%      |

---

### 12.2 アクション

| 条件      | 対応                    |
| --------- | ----------------------- |
| API負荷増 | API追加                 |
| Reco遅延  | Reco追加                |
| DB遅延    | インデックス / レプリカ |

---

## 13. データスケール対応

---

### 13.1 データ増加

| 項目        | 対応                |
| ----------- | ------------------- |
| 商品数増    | embeddingキャッシュ |
| ログ増      | partition / TTL     |
| interaction | 集計BT化            |

---

## 14. スケール段階設計（重要）

---

### Phase1（MVP）

```
API 1台
Reco 1台
DB 1台
Cache 任意
```

---

### Phase2

```
API 複数
Reco 複数
Redis導入
```

---

### Phase3

```
DB分離
Redis Cluster
LB導入
```

---

## 15. 設計制約

```
・セッション禁止（スケール阻害）
・状態はDBへ
・同期処理は軽量化
```

---

## 16. 設計原則まとめ

```
・スケールは「後から拡張」前提
・まずはシンプルに
・ボトルネックを分離
・ReCoは独立スケール
```

---

## 17. 一言まとめ

```
MVPでは単一構成で開始し、
負荷増加に応じてAPI・Recoを水平スケールし、
キャッシュとDB分離で性能を維持する
```

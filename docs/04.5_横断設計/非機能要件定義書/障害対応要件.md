## 1. 目的

本ドキュメントは、本サービスにおける障害の検知・対応・回復・影響最小化のための要件を定義し、

API設計・Reco設計・基盤設計・運用設計における具体的な実装指針とする。

---

## 2. 対象範囲

| 対象     | 内容            |
| -------- | --------------- |
| API      | 外部公開API     |
| Reco     | レコメンド処理  |
| DB       | データアクセス  |
| Cache    | Redis / メモリ  |
| 外部依存 | 将来の外部API等 |

---

## 3. 障害の分類

### 3.1 障害カテゴリ

| 種別                 | 内容                       |
| -------------------- | -------------------------- |
| アプリケーション障害 | バグ・例外                 |
| インフラ障害         | サーバダウン、ネットワーク |
| 性能劣化             | タイムアウト、遅延         |
| データ障害           | 不整合、欠損               |
| 外部依存障害         | 外部API障害                |

---

### 3.2 影響レベル

| レベル | 定義                           |
| ------ | ------------------------------ |
| L1     | サービス全停止                 |
| L2     | 主要機能停止（レコメンド不可） |
| L3     | 一部機能劣化（遅い、精度低下） |
| L4     | 軽微影響                       |

---

## 4. 障害検知要件

### 4.1 検知対象

| 指標           | 閾値      |
| -------------- | --------- |
| APIエラー率    | 5%以上    |
| レスポンス時間 | p95 > 3秒 |
| タイムアウト率 | 1%以上    |
| Reco失敗率     | 3%以上    |

---

### 4.2 検知方法

```
・メトリクス監視（latency / error_rate）
・ログ監視（error_log）
・ヘルスチェック（/health）
```

---

### 4.3 検知タイミング

| 項目     | 要件    |
| -------- | ------- |
| 検知遅延 | 1分以内 |
| 通知     | 即時    |

---

## 5. リトライ要件

### 5.1 リトライ対象

| 対象        | 条件             |
| ----------- | ---------------- |
| Reco内部API | timeout / 5xx    |
| DBアクセス  | 一時的接続エラー |
| Cache       | 接続失敗         |

---

### 5.2 リトライ仕様

| 項目   | 内容                        |
| ------ | --------------------------- |
| 回数   | 最大2回                     |
| 間隔   | 100ms → 300ms               |
| 方式   | exponential backoff         |
| 対象外 | バリデーションエラー（4xx） |

---

### 5.3 実装責務

| レイヤ | 内容                   |
| ------ | ---------------------- |
| API    | Reco呼び出しのリトライ |
| Reco   | DBアクセスのリトライ   |

---

## 6. タイムアウト要件

### 6.1 設定値

| 対象     | タイムアウト |
| -------- | ------------ |
| API全体  | 5秒          |
| Reco内部 | 4秒          |
| 各処理   | 1秒          |

---

### 6.2 挙動

```
・timeout時は即座に処理中断
・run_event_logに記録
・フォールバックへ遷移
```

---

## 7. フォールバック要件（重要）

### 7.1 基本方針

```
・必ず何かしらの結果を返す
・空レスポンスは禁止
```

---

### 7.2 フォールバックパターン

| 障害箇所        | フォールバック   |
| --------------- | ---------------- |
| Meaning推定失敗 | デフォルト特徴量 |
| Retrieval失敗   | 人気商品返却     |
| Ranking失敗     | Retrieval順返却  |
| 全体失敗        | 固定おすすめ返却 |

---

### 7.3 UX要件

| 項目         | 内容                       |
| ------------ | -------------------------- |
| ユーザー表示 | 「おすすめを表示しました」 |
| エラー露出   | 内部エラー非表示           |
| 再試行       | UIから可能                 |

---

## 8. graceful degradation

### 8.1 定義

機能を段階的に落としながらサービス継続

---

### 8.2 適用

| 状態     | 挙動             |
| -------- | ---------------- |
| 高負荷   | ranking簡略化    |
| DB遅延   | キャッシュ優先   |
| 部分障害 | 一部機能スキップ |

---

## 9. 障害記録要件

### 9.1 必須ログ

| ログ          | 内容       |
| ------------- | ---------- |
| run_event_log | 各ステップ |
| error_log     | エラー内容 |
| metric_log    | 指標       |

---

### 9.2 記録内容

```
・runId
・処理ステップ
・エラー種別
・処理時間
・再試行回数
```

---

## 10. 再現性要件（重要）

```
・run単位で再実行可能
・入力条件を完全保存
・使用config/version記録
```

---

## 11. 復旧要件

### 11.1 自動復旧

| 項目           | 内容 |
| -------------- | ---- |
| コンテナ再起動 | 自動 |
| 接続再試行     | 自動 |

---

### 11.2 手動復旧

| 項目         | 内容 |
| ------------ | ---- |
| ロールバック | 可能 |
| config切替   | 即時 |

---

## 12. 障害影響の最小化

### 12.1 分離設計

```
API と Reco を分離
```

---

### 12.2 影響範囲制御

| 対象           | 制御       |
| -------------- | ---------- |
| Reco障害       | APIは生存  |
| キャッシュ障害 | 通常処理へ |

---

## 13. SLO（障害観点）

| 指標             | 目標  |
| ---------------- | ----- |
| 可用性           | 99.0% |
| エラー率         | < 1%  |
| フォールバック率 | < 5%  |

---

## 14. 設計原則

```
・失敗を前提に設計する
・必ず結果を返す
・ログで完全再現可能にする
・部分失敗を許容する
・依存を最小化する
```

---

## 15. 一言まとめ

```
障害は必ず起きる前提で、
リトライ・フォールバック・分離により
ユーザー影響を最小化しつつ、
再現可能な設計とする
```

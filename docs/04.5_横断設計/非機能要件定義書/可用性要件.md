## 1. 目的

本ドキュメントは、本サービスの可用性（Availability）に関する要件を定義し、

サービス停止・劣化時の影響を最小化し、継続的に利用可能な状態を維持するための設計指針とする。

---

## 2. 対象範囲

| 対象     | 内容                    |
| -------- | ----------------------- |
| API      | 外部公開API             |
| Reco     | 推薦処理                |
| DB       | データストア            |
| Cache    | Redis / メモリ          |
| インフラ | コンテナ / ネットワーク |

---

## 3. 可用性指標（SLO / SLA）

---

### 3.1 SLO（内部目標）

| 指標             | 目標値                          |
| ---------------- | ------------------------------- |
| 可用性           | 99.0%以上                       |
| API成功率        | 99.0%以上                       |
| レコメンド成功率 | 97.0%以上（フォールバック含む） |
| フォールバック率 | 5%以下                          |

---

### 3.2 SLA（外部保証）

| 指標   | 値                        |
| ------ | ------------------------- |
| 可用性 | MVPでは未保証（将来設定） |

---

## 4. 可用性設計方針

```
・完全停止を避ける（partial failure許容）
・フォールバックで機能を維持する
・単一障害点（SPOF）を最小化する
・障害時もAPIは応答する
```

---

## 5. 冗長化設計

---

### 5.1 コンポーネント別方針

| コンポーネント | MVP方針                  | 将来             |
| -------------- | ------------------------ | ---------------- |
| API            | 単一インスタンス         | 複数インスタンス |
| Reco           | 単一インスタンス         | 水平スケール     |
| DB             | 単一（バックアップあり） | レプリケーション |
| Cache          | 単一                     | Redis Cluster    |

---

### 5.2 冗長化レベル

| レベル | 対応         |
| ------ | ------------ |
| MVP    | 再起動ベース |
| 将来   | Multi-AZ     |

---

## 6. 単一障害点（SPOF）対策

---

### 6.1 現状（MVP）

| 対象  | 対策             |
| ----- | ---------------- |
| API   | 自動再起動       |
| Reco  | APIと分離        |
| DB    | 定期バックアップ |
| Cache | 障害時は無視     |

---

### 6.2 将来対策

```
・DBレプリケーション
・API/Reco複数台構成
・ロードバランサ導入
```

---

## 7. 障害時の継続動作

---

### 7.1 基本方針

```
・完全停止より品質劣化を優先
```

---

### 7.2 コンポーネント別挙動

| 障害箇所     | 挙動               |
| ------------ | ------------------ |
| Reco停止     | APIはエラー返却    |
| Reco部分失敗 | フォールバック実行 |
| DB遅延       | キャッシュ優先     |
| Cache停止    | 通常処理へ         |

---

## 8. 復旧要件

---

### 8.1 目標値

| 指標              | 値        |
| ----------------- | --------- |
| RTO（復旧時間）   | 1時間以内 |
| RPO（データ損失） | 1時間以内 |

---

### 8.2 復旧方法

| 種別         | 方法                 |
| ------------ | -------------------- |
| コンテナ障害 | 自動再起動           |
| アプリ障害   | デプロイし直し       |
| DB障害       | バックアップから復元 |

---

---

## 11. データ可用性

---

### 11.1 バックアップ

| 項目 | 内容    |
| ---- | ------- |
| 頻度 | 1日1回  |

---

### 11.2 データ保護

```
・run / interaction は重要データ

```

---

## 12. キャッシュと可用性

---

### 12.1 方針

```
・キャッシュは必須依存にしない
```

---

### 12.2 挙動

| 状態           | 挙動 |
| -------------- | ---- |
| キャッシュ正常 | 使用 |
| キャッシュ障害 | 無視 |

---

## 13. 可用性の測定

---

### 13.1 指標

| 指標          | 内容             |
| ------------- | ---------------- |
| uptime        | 稼働率           |
| error_rate    | エラー率         |
| fallback_rate | フォールバック率 |

---

## 14. API/Reco分離の影響

---

### 14.1 メリット

```
・障害分離
・部分停止可能
```

---

### 14.2 挙動

| 状態               | 結果               |
| ------------------ | ------------------ |
| API正常 / Reco異常 | エラー or fallback |
| API異常            | 全停止             |

---

## 15. 設計原則

```
・止めないことを最優先
・壊れても戻せる設計
・依存を減らす
・単純に保つ（MVP）
```

---

## 16. 一言まとめ

```
MVPでは単一構成を前提としつつ、
障害時はフォールバックと再起動で継続し、
将来は冗長化により可用性を向上させる
```

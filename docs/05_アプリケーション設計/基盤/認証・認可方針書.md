## 1. 目的

本ドキュメントは、本サービスにおける認証（Authentication）および認可（Authorization）の方針を定義し、

以下の設計・実装・運用の前提を明確化することを目的とする。

- API設計
- 画面設計
- セキュリティ設計
- 権限設計
- 監査ログ設計

---

## 2. スコープ

| 対象         | 内容                 |
| ------------ | -------------------- |
| クライアント | ブラウザ             |
| Webサーバ    | フロントエンドサーバ |
| API          | 外部公開API          |
| Reco         | 内部サービス         |
| 管理画面     | 管理者向けUI         |
| 評価画面     | 評価者向けUI         |

---

## 3. 基本方針

### 3.1 認証方針

```
・認証はJWT（Bearer Token）を用いたステートレス認証とする
・APIサーバはセッション状態を保持しない
・認証情報はクライアントが保持し、API呼び出しごとに送信する
```

### 3.2 認可方針

```
・認可はロールベースを基本とする
・必要に応じてリソース所有者チェックを組み合わせる
・最小権限の原則を適用する
```

### 3.3 サービス間認証方針

```
・API ⇄ Reco 間は内部認証を必須とする
・内部サービスは外部公開しない
```

---

## 4. 認証方式

### 4.1 採用方式

| 項目        | 方針                |
| ----------- | ------------------- |
| 方式        | JWT Bearer Token    |
| セッション  | 不採用              |
| API認証     | Authorizationヘッダ |
| 内部API認証 | サービス間トークン  |

---

### 4.2 採用理由

| 観点             | 理由                       |
| ---------------- | -------------------------- |
| スケーラビリティ | APIをステートレスにできる  |
| 可用性           | セッション共有ストア不要   |
| API/Reco分離     | サービス間連携と相性が良い |
| 運用             | 水平スケールしやすい       |

---

## 5. 認証主体

### 5.1 外部認証主体

| 主体         | 説明             |
| ------------ | ---------------- |
| 一般ユーザー | レコメンド利用者 |
| 管理者       | 管理画面利用者   |
| 評価者       | human_eval実施者 |

### 5.2 内部認証主体

| 主体         | 説明             |
| ------------ | ---------------- |
| APIサービス  | Reco呼び出し主体 |
| Recoサービス | 内部処理主体     |

---

## 6. トークン方針

### 6.1 トークン種別

| トークン             | 用途            |
| -------------------- | --------------- |
| アクセストークン     | APIアクセス     |
| 内部サービストークン | API ⇄ Reco 認証 |

---

### 6.2 アクセストークン要件

| 項目         | 要件                  |
| ------------ | --------------------- |
| 送信方法     | Authorization: Bearer |
| 有効期限     | 1時間（MVP）          |
| 署名検証     | 必須                  |
| 有効期限検証 | 必須                  |
| issuer検証   | 必須                  |
| audience検証 | 必須                  |

---

### 6.3 JWTクレーム方針

| クレーム | 内容           | 必須 |
| -------- | -------------- | ---- |
| sub      | ユーザー識別子 | 必須 |
| iss      | 発行者         | 必須 |
| aud      | 対象API        | 必須 |
| exp      | 有効期限       | 必須 |
| iat      | 発行時刻       | 推奨 |
| role     | ロール         | 必須 |
| tenant   | 将来拡張用     | 任意 |

---

### 6.4 JWTに含めないもの

| 項目             | 理由           |
| ---------------- | -------------- |
| 個人情報の詳細   | 漏えいリスク   |
| 大きな権限一覧   | トークン肥大化 |
| 頻繁に変わる状態 | 失効しづらい   |

---

## 7. ロール設計

### 7.1 ロール一覧

| ロール    | 説明         |
| --------- | ------------ |
| user      | 一般ユーザー |
| admin     | 管理者       |
| evaluator | 評価者       |

---

### 7.2 ロールの責務

| ロール    | 主責務                                         |
| --------- | ---------------------------------------------- |
| user      | 自身の推薦実行・履歴参照・行動記録             |
| admin     | 管理対象データの全件参照・設定変更・モデル運用 |
| evaluator | 評価タスク参照・評価結果登録                   |

---

## 8. 認可モデル

### 8.1 基本モデル

```
認可 = ロール判定 + リソース所有者判定
```

---

### 8.2 認可の考え方

| パターン         | 内容                                    |
| ---------------- | --------------------------------------- |
| ロールベース認可 | admin / evaluator / user の区分で制御   |
| 所有者認可       | user は自分のrun / resultのみ参照可能   |
| 内部認可         | 内部APIはサービス間認証済み主体のみ許可 |

---

## 9. リソース別認可方針

### 9.1 一般ユーザー系

| リソース               | user           | admin  | evaluator |
| ---------------------- | -------------- | ------ | --------- |
| recommendations        | 実行可         | 可     | 不要      |
| interactions           | 自分のみ可     | 可     | 不要      |
| recommendation_runs    | 自分のみ参照可 | 全件可 | 不要      |
| recommendation_results | 自分のみ参照可 | 全件可 | 不要      |

---

### 9.2 管理系

| リソース         | user | admin | evaluator |
| ---------------- | ---- | ----- | --------- |
| semantic_configs | 不可 | 可    | 不可      |
| model_versions   | 不可 | 可    | 不可      |
| run_event_log    | 不可 | 可    | 不可      |

---

### 9.3 評価系

| リソース     | user | admin | evaluator |
| ------------ | ---- | ----- | --------- |
| eval_tasks   | 不可 | 可    | 可        |
| eval_results | 不可 | 可    | 可        |

---

## 10. API種別ごとの認可方針

### 10.1 一般API

| API                                     | 認可方針         |
| --------------------------------------- | ---------------- |
| POST /recommendations                   | userのみ         |
| POST /interactions                      | userのみ         |
| GET /recommendation-runs                | 自分のデータのみ |
| GET /recommendation-runs/{runId}        | 自分のrunのみ    |
| GET /recommendation-runs/{runId}/result | 自分のrunのみ    |

---

### 10.2 管理API

| API        | 認可方針  |
| ---------- | --------- |
| /admin/... | adminのみ |

---

### 10.3 評価API

| API                | 認可方針         |
| ------------------ | ---------------- |
| GET /eval/tasks    | evaluator, admin |
| POST /eval/results | evaluator, admin |
| GET /eval/results  | evaluator, admin |

---

### 10.4 内部API

| API                | 認可方針        |
| ------------------ | --------------- |
| /internal/reco/... | APIサービスのみ |

---

## 11. 所有者判定方針

### 11.1 対象

| リソース              | 判定方法                              |
| --------------------- | ------------------------------------- |
| recommendation_run    | run.owner_user_id = token.sub         |
| recommendation_result | run経由で所有者判定                   |
| interaction           | interaction.owner_user_id = token.sub |

---

### 11.2 方針

```
userロールは、自身が作成・実行したリソースにのみアクセスできる
```

---

## 12. 認証・認可エラー方針

### 12.1 ステータスコード

| ケース             | ステータス |
| ------------------ | ---------- |
| 認証情報なし       | 401        |
| トークン不正       | 401        |
| 権限不足           | 403        |
| 所有者不一致       | 403        |
| 存在しないリソース | 404        |

---

### 12.2 返却方針

```
・詳細すぎる内部情報は返さない
・ログには詳細を残す
```

---

## 13. 内部サービス認証

### 13.1 API ⇄ Reco

| 項目     | 方針                        |
| -------- | --------------------------- |
| 認証方式 | サービス間トークン          |
| 送信方法 | 専用ヘッダ                  |
| 検証     | APIサービス発行主体のみ許可 |

---

### 13.2 方針

```
Recoサービスは外部から直接呼び出せない構成とする
```

---

## 14. クライアント側の扱い

### 14.1 ブラウザ側状態

| 項目             | 方針             |
| ---------------- | ---------------- |
| アクセストークン | クライアント保持 |
| UI状態           | クライアント保持 |
| 真のデータ       | サーバ保持       |

---

### 14.2 原則

```
クライアントは認証情報を提示するだけであり、
認可判定の主体は常にサーバ側とする
```

---

## 15. 監査・ログ方針

### 15.1 記録対象

| 項目                   | 記録 |
| ---------------------- | ---- |
| ログイン成功           | 記録 |
| ログイン失敗           | 記録 |
| 認可失敗               | 記録 |
| 重要操作（設定変更等） | 記録 |

---

### 15.2 禁止事項

| 項目               | 方針 |
| ------------------ | ---- |
| JWT本体の全文出力  | 禁止 |
| パスワード出力     | 禁止 |
| 個人情報の過剰出力 | 禁止 |

---

## 16. 将来拡張方針

| 項目                 | 将来対応            |
| -------------------- | ------------------- |
| リフレッシュトークン | 対応可能            |
| MFA                  | 管理者向け導入候補  |

---

## 17. 設計原則まとめ

```
・認証はステートレス
・認可は最小権限
・userは自分のデータのみ
・adminは管理範囲のみ
・内部APIは外部遮断
```

---

## 18. 一言まとめ

```
本サービスでは、JWTによるステートレス認証を採用し、
ロールベース認可と所有者判定を組み合わせることで、
スケーラブルかつ安全なアクセス制御を実現する
```

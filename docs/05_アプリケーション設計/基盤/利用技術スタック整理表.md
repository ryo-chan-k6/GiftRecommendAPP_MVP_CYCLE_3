## 1. 目的

本ドキュメントは、本サービスの技術スタックを整理し、

- 採用技術の全体像の可視化
- コンポーネントごとの責務と技術選定理由の明確化
- 開発・運用・拡張の前提統一

を目的とする。

---

## 2. スコープ

| レイヤ        | 対象          |
| ------------- | ------------- |
| フロント      | Webアプリ     |
| アプリ        | API / Reco    |
| データ        | DB / Cache    |
| バッチ        | ETL / 集約    |
| 外部依存      | Embedding API |
| インフラ      | 実行基盤      |
| Observability | 監視・ログ    |

---

# 3. 全体構成（俯瞰）

```
[Browser]
   ↓
[Next.js Web]
   ↓
[API (Node.js)]
   ↓
[Reco (Python)]
   ↓
[PostgreSQL / Redis]

Batch:
[Python ETL / Feature / Embedding / Aggregation]

External:
[Embedding API]
```

---

# 4. レイヤ別技術スタック

---

## 4.1 フロントエンド

| 項目           | 技術               |
| -------------- | ------------------ |
| フレームワーク | Next.js            |
| 言語           | TypeScript         |
| UI             | React              |
| 通信           | fetch / axios      |
| 状態管理       | React state（MVP） |
| 認証           | JWT（Bearer）      |

### 選定理由

- SSR/CSR両対応
- API連携が容易
- MVP開発速度が高い

---

## 4.2 APIサービス

| 項目             | 技術             |
| ---------------- | ---------------- |
| ランタイム       | Node.js          |
| フレームワーク   | Express          |
| 言語             | TypeScript       |
| API仕様          | OpenAPI          |
| クライアント生成 | orval            |
| DBアクセス       | supabase-js / pg |
| 認証             | JWT              |

---

### 責務

- 外部API提供
- 認証 / 認可
- recommendation_run / result 管理
- Reco呼び出し

---

### 選定理由

- フロントとの親和性（TypeScript統一）
- 軽量で柔軟
- OpenAPI連携が容易

---

## 4.3 Recoサービス

| 項目           | 技術                   |
| -------------- | ---------------------- |
| ランタイム     | Python                 |
| フレームワーク | FastAPI                |
| 言語           | Python                 |
| 数値計算       | numpy / pandas         |
| ML/embedding   | OpenAI API 等          |
| ベクトル検索   | pgvector（PostgreSQL） |

---

### 責務

- 入力解析
- 意味推定（user_feature）
- retrieval（embedding）
- ranking（feature計算）

---

### 選定理由

- 数値・ML処理に強い
- ライブラリが豊富
- 実験・改善サイクルが速い

---

## 4.4 データベース

| 項目         | 技術       |
| ------------ | ---------- |
| DB           | PostgreSQL |
| ホスティング | Neon       |
| 拡張         | pgvector   |

---

### データ種別

| 種別      | 内容                   |
| --------- | ---------------------- |
| 正本      | run / result / config  |
| 行動      | interaction            |
| 派生      | feature / stats / eval |
| embedding | item_embedding（永続） |

---

### 選定理由

- RDBとしての安定性
- pgvector対応
- SaaSとして軽量（Neon）

---

## 4.5 キャッシュ

| 項目       | 技術  |
| ---------- | ----- |
| キャッシュ | Redis |

---

### 利用用途

| 対象           | 内容           |
| -------------- | -------------- |
| item_embedding | 高速参照       |
| user_embedding | 短期キャッシュ |
| retrieval結果  | 一時キャッシュ |

---

### 選定理由

- 低レイテンシ
- TTL管理が容易
- 分散対応可能

---

## 4.6 バッチ処理

| 項目 | 技術               |
| ---- | ------------------ |
| 言語 | Python             |
| 実行 | CLI / スケジューラ |
| HTTP | httpx              |
| DB   | psycopg            |

---

### 処理内容

| ジョブ      | 内容                     |
| ----------- | ------------------------ |
| ETL         | 商品データ取得           |
| Feature     | item_feature生成         |
| Embedding   | item_embedding生成       |
| Stats       | 分布統計                 |
| Aggregation | interaction / metric集約 |
| Evaluation  | offline_eval / FB分類    |

---

### 選定理由

- PythonでRecoと統一
- バッチ処理が書きやすい
- データ処理に適している

---

## 4.7 外部依存

# 6. 技術選定の全体方針

```
・フロント/APIはTypeScriptで統一
・Reco/BatchはPythonで統一
・DBはPostgreSQLに集約
・キャッシュはRedisで補助
・外部APIは最小限に依存
・MVPはシンプル構成で開始
・将来のスケールを阻害しない選定を行う
```

---

# 7. MVPでやりすぎないこと

```
・Kubernetes導入
・複雑なマイクロサービス分割
・分散トレーシングの完全実装
・データレイク構築
・リアルタイムストリーミング基盤
```

# 9. リスクと対策

| リスク          | 対策                 |
| --------------- | -------------------- |
| Embeddingコスト | item_embedding永続化 |
| DB肥大化        | 集約 / 保持期間制御  |
| Reco負荷        | キャッシュ / 分離    |
| API遅延         | Redis活用            |
| 分布崩壊        | Observability        |

---

# 10. 設計原則まとめ

```
・責務単位で技術を分離する
・データはPostgreSQLに集約する
・計算負荷はRecoに閉じる
・高速化はRedisで補助する
・監視は全レイヤ対象とする
・MVPでは最小構成にする
```

## 1. 目的

本書は、本システムにおけるログ設計の基本方針を定義し、以下を明確化することを目的とする。

- ログの役割と責務
- ログのドメイン単位での管理方針
- ログの分類および保存対象
- 状態管理・データ設計との役割分担
- 観測・評価・改善への接続

---

## 2. 適用範囲

### 対象

- 推薦実行ログ
- ユーザー行動ログ
- 評価関連ログ
- メトリクスログ

### 非対象

- 状態管理（→ 状態遷移設計書）
- 正本データ（→ テーブル設計）
- 集約済み統計（→ 派生テーブル）

---

## 3. 設計前提

### 3-1. ログの役割

**事実**

- 本システムは「レコメンドの再現性」と「改善サイクル」を重視する

**推論**

- ログは以下の役割を持つ

| 役割     | 内容                       |
| -------- | -------------------------- |
| トレース | 実行の追跡・障害調査       |
| 観測     | 行動分析・改善             |
| 評価基盤 | 評価・比較・検証の元データ |

---

### 3-2. ログと他データの関係

| 種類 | 役割                 |
| ---- | -------------------- |
| 状態 | トランザクション管理 |
| 正本 | 業務データ           |
| ログ | イベント・履歴       |
| 派生 | 集約・分析           |

---

## 4. ログ設計の基本原則

### 4-1. イベント単位で記録する

- 状態ではなく「発生事実」を記録する
- 同一対象に複数イベントを許容

---

### 4-2. トランザクション単位で紐づける

- `recommendation_run_id` を中心とする
- 必要に応じて item / task に紐づける

---

### 4-3. ログは再現性を補助するものとする

- ログがなくても再計算可能な設計を維持
- ログは補助情報

---

### 4-4. ログは削除・集約前提で設計する

- 生ログは短期〜中期保存
- 長期利用は集約テーブルへ

---

### 4-5. ログは責務単位で管理する

（本設計のコア）

---

## 5. ログ統合方針（重要）

### 5-1. 基本方針

本システムでは、ログを単一の汎用 `event_log` に統合せず、

**ドメイン責務単位で統合管理する**

---

### 5-2. 理由

**事実**

- ログには業務意味の異なるイベントが混在する

**推論**

- 完全統合すると以下の問題が発生する

| 問題           | 内容                     |
| -------------- | ------------------------ |
| 可読性低下     | event_type依存が強くなる |
| スキーマ肥大化 | payload_json依存         |
| 責務曖昧       | 業務境界が不明確         |
| 拡張難易度     | 変更影響が広がる         |

---

### 5-3. 採用方針

👉 **ドメイン単位でログを統合する**

---

## 6. ドメイン別ログ設計方針

---

### 6-1. 推薦実行ドメインログ

### テーブル

`recommendation_run_event_log`

### 統合対象

- phase_log
- error_log

### 役割

- 実行トレース
- 障害調査

### 主なイベント

- run_started
- phase_started
- phase_completed
- run_failed
- run_aborted

### 特徴

- run単位で完全に紐づく
- 時系列イベント

---

### 6-2. ユーザー行動ドメインログ

### テーブル

`user_interaction_log`

### 統合対象

- view_log
- click_log

### 役割

- ユーザー行動の記録

### 主なイベント

- item_viewed
- item_clicked

### 特徴

- 高頻度
- 分析の基礎データ

---

### 6-3. 推薦評価ドメイン

### テーブル

`recommendation_feedback`

### 方針

- ログとして統合せず**独立保持**

### 理由

- 明示評価の事実
- 正本として扱う
- 業務データに近い

---

### 6-4. 人手評価ドメイン

### テーブル

- `human_eval_task`（状態管理）
- `human_eval_result`（評価記録）

### 方針

- ログ統合しない

### 理由

- 評価結果は業務データ
- 状態管理対象が存在

---

### 6-5. 観測・分析ドメイン

### テーブル

- `metric_log`

### 方針

- 独立維持

### 理由

- 高頻度・数値中心
- 分布分析用途

---

## 7. トレーサビリティ設計

### 7-1. キー設計

| キー                          | 用途     |
| ----------------------------- | -------- |
| recommendation_run_id         | 実行単位 |
| recommendation_result_item_id | 商品単位 |
| human_eval_task_id            | 評価単位 |

---

### 7-2. トレース構造

```
recommendation_run
   ├── recommendation_run_event_log
   ├── metric_log
   └── recommendation_result_item
           ├── user_interaction_log
           ├── recommendation_feedback
           └── human_eval_task
                   └── human_eval_result
```

---

## 8. ログ粒度設計

| ログ            | 粒度               |
| --------------- | ------------------ |
| run_event_log   | phase / エラー単位 |
| interaction_log | 1操作単位          |
| metric_log      | item × feature     |

| ---------- | -------- |

# 一言まとめ

- ログは「イベント・履歴」を記録する

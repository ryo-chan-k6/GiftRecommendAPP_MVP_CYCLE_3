## 1. 目的・スコープ

本ドキュメントは、Gift Recommendation Service における API 設計の標準ルールを定義することを目的とする。

本標準は以下を対象とする。

- 外部公開API（Webクライアント向け）
- 管理API（Admin / 評価運用）
- 内部連携API（API → Reco 等）

本標準は、API仕様書作成時の共通ルールとして利用し、**設計の一貫性・保守性・拡張性を担保する**。

本サービスにおけるAPIリソースは、「リソース定義書」に基づいて定義する。
URL Pathは当該リソース定義に従って設計する。


## 3. URL / リクエスト構造設計方針

### 3-1. 基本原則

HTTPリクエストの各要素は、以下の役割に限定して使用する。

| 要素            | 役割                                     |
| --------------- | ---------------------------------------- |
| URL Path        | リソースの所在・階層                     |
| Path Parameter  | 特定リソースの識別子                     |
| Query Parameter | 参照条件・絞り込み・並び替え・ページング |
| Header          | 認証、トレーシング、冪等性などの横断情報 |
| Request Body    | 作成内容、実行条件、複雑な入力条件       |

### 3-2. 原則

- URLは「何を対象にしているか」を表す
- Query は「どう取得したいか」を表す
- Body は「何を実行・作成したいか」を表す
- Header は「この通信をどう扱うか」を表す

## 4. URL設計方針

### 4-1. 基本ルール

| 項目       | ルール               |
| ---------- | -------------------- |
| ベースパス | `/api/v1/...`        |
| 命名       | 名詞ベース           |
| 複数形     | コレクションは複数形 |
| 区切り     | kebab-case           |
| 階層       | 必要最小限           |
| 動詞       | 原則使用しない       |

### 4-2. API区分

| 区分    | パス                      |
| ------- | ------------------------- |
| 一般API | `/api/v1/...`             |
| 管理API | `/api/v1/admin/...`       |
| 評価API | `/api/v1/eval/...`        |
| 実験API | `/api/v1/experiments/...` |

### 4-3. 例

```
/api/v1/recommendations
/api/v1/interactions
/api/v1/admin/semantic-configs
/api/v1/eval/tasks
```

## 5. Path設計方針

### 5-1. Path の意味

Path は、API が操作対象とする **リソース種別と階層** を表す。

### 5-2. Path に含めてよい情報

| 含めてよいもの       | 例                        |
| -------------------- | ------------------------- |
| リソース種別         | `/recommendations`        |
| サブカテゴリ         | `/admin/semantic-configs` |
| リソース識別子       | `/{runId}`                |
| 業務操作名（例外的） | `/{configId}/activate`    |

### 5-3. 業務操作名（例外的）の定義

業務操作名とは、REST の一般原則では名詞中心としつつも、**状態変更や業務操作の意味を明示する必要がある場合に限り、例外的に Path の末尾に付与する操作名**を指す。

例：

- `/{configId}/activate`
- `/{configId}/deactivate`
- `/{runId}/retry`

### 5-4. Path に含めない情報

| 含めないもの         | 理由                       |
| -------------------- | -------------------------- |
| 検索条件             | Query または Body に置く   |
| ソート条件           | 取得方法であり識別ではない |
| ページ番号           | 同上                       |
| 認証情報             | Header で扱う              |
| 可変な内部パラメータ | 契約が不安定になる         |

---

## 6. Path Parameter 設計方針

### 6-1. 利用目的

Path Parameter は、**単一リソースを一意に識別するためにのみ使用する**。

### 6-2. 利用ルール

| ルール   | 内容                                                                    |
| -------- | ----------------------------------------------------------------------- |
| 用途限定 | ID表現に限定する                                                        |
| 一意性   | その path 配下で一意であること                                          |
| 命名     | `{runId}`, `{configId}`, `{taskId}` のように camelCase の識別子名を使う |
| 数       | 必要最小限にする                                                        |
| 意味     | 「どのリソースか」が明確であること                                      |

### 6-3. 使用例

| 用途                       | 例                                          |
| -------------------------- | ------------------------------------------- |
| recommendation_run詳細参照 | `/api/v1/admin/recommendation-runs/{runId}` |
| semantic_config詳細参照    | `/api/v1/admin/semantic-configs/{configId}` |
| human_eval_task詳細参照    | `/api/v1/eval/tasks/{taskId}`               |

### 6-4. 使用禁止例

| NG例                        | 理由                                          |
| --------------------------- | --------------------------------------------- |
| `/recommendations/{userId}` | recommendation 実行条件であり識別対象ではない |
| `/interactions/{type}`      | type は識別子ではなく分類条件                 |
| `/runs/{date}`              | date は一意識別子ではない                     |

---

## 7. Query Parameter 設計方針

### 7-1. 利用目的

Query Parameter は、**参照系 API の取得条件** にのみ使用する。

### 7-2. Query に載せる対象

| 対象         | 例                          |
| ------------ | --------------------------- |
| 一覧絞り込み | `status=active`             |
| 検索文字列   | `keyword=flower`            |
| ソート条件   | `sort=createdAt&order=desc` |
| ページング   | `page=1&pageSize=20`        |
| 参照モード   | `includeDetails=true`       |

### 7-3. Query に載せない対象

| 対象                             | 理由                           |
| -------------------------------- | ------------------------------ |
| 複雑な検索条件の配列・ネスト構造 | URL可読性と保守性が落ちる      |
| 実行系APIの入力本体              | 取得条件ではなく実行条件だから |
| 長文テキスト                     | URLに不向き                    |
| 秘匿性が必要な情報               | URLはログ等に残りやすい        |

### 7-4. 適用ルール

- 原則として **GET のみで使用する**
- **参照条件が単純な場合のみ Query を使う**
- 複雑条件は POST + Body に切り替える

### 7-5. 使用例

```
GET /api/v1/admin/recommendation-runs?page=1&pageSize=20&sort=createdAt&order=desc
GET /api/v1/admin/semantic-configs?status=active
GET /api/v1/eval/tasks?assigneeId=U001&status=open
```

## 8. HTTPヘッダー設計方針

### 8-1. Header の役割

Header は、**リソース内容そのものではなく、通信や横断制御に関する情報** を表す。

### 8-2. 基本ヘッダー

| ヘッダー      | 用途               |
| ------------- | ------------------ |
| Authorization | 認証               |
| Content-Type  | `application/json` |
| Accept        | `application/json` |
| X-Request-Id  | リクエスト追跡     |

### 8-3. オプション

| ヘッダー        | 用途        |
| --------------- | ----------- |
| Idempotency-Key | 重複防止    |
| X-Service-Auth  | 内部API認証 |

### 8-4. Header に置かない対象

| 対象           | 理由                    |
| -------------- | ----------------------- |
| 検索条件       | Query / Body に置くべき |
| 業務データ本体 | Body に置くべき         |
| 業務状態       | URL / Body で表現すべき |

## 9. Request / Response Body 設計方針

### 9-1. 共通ルール

| 項目         | ルール                       |
| ------------ | ---------------------------- |
| フォーマット | JSON                         |
| 命名         | camelCase                    |
| 日時         | ISO8601                      |
| ID           | string                       |
| null         | 明示的に許可する場合のみ使用 |

### 9-2. Request Body の利用目的

Request Body は、**作成内容・実行条件・複雑な入力条件** を表すために使用する。

### 9-3. Body に載せる対象

| 対象           | 例                         |
| -------------- | -------------------------- |
| 作成内容       | semantic_config 登録内容   |
| 実行条件       | recommendation 実行条件    |
| 記録内容       | interaction の内容         |
| 複雑な検索条件 | 複数条件、配列、ネスト構造 |
| 長文入力       | ユーザー自由入力文         |

### 9-4. Body に載せない対象

| 対象               | 理由              |
| ------------------ | ----------------- |
| 単純な一覧取得条件 | Query で十分      |
| 認証情報           | Header で扱う     |
| リソース識別子のみ | Path で表現すべき |

### 9-5. Response 設計

| 項目     | 方針                       |
| -------- | -------------------------- |
| UI最適化 | UIで使いやすい形式         |
| 内部情報 | 原則非公開                 |
| 追跡ID   | runId / requestId を含める |

## 10. Query Parameter と Request Body の使い分け

### 10-1. 判断基準

| 条件                               | 配置先 |
| ---------------------------------- | ------ |
| 単純な参照条件                     | Query  |
| 一覧の絞り込み・ソート・ページング | Query  |
| 複雑な検索条件                     | Body   |
| 長文自由入力                       | Body   |
| 実行系の条件                       | Body   |

### 10-2. Query を使うケース

以下を満たすときは Query を使う。

- GET である
- 条件が単純
- URLで見ても理解しやすい
- 条件数が少ない
- 配列・ネストが中心ではない

例：

```
GET /api/v1/admin/recommendation-runs?status=success&page=1&pageSize=20
GET /api/v1/eval/tasks?status=open&assigneeId=U001
```

### 10-3. Body を使うケース

以下のいずれかを満たすときは Body を使う。

- POST である
- 検索条件が複雑
- 配列やネストを含む
- 長文入力を含む
- 実行入力そのもの

例：

```
POST /api/v1/recommendations
{
  "recipient": {
    "relationship":"boss"
  },
  "occasion": {
    "type":"birthday"
  },
  "budget": {
    "min":3000,
    "max":5000
  },
  "preferredFeatures": [
"上品",
"落ち着いた"
  ],
  "freeText":"無難だけど少し印象に残るもの"
}
```

### 10-4. サービス適用ルール

- レコメンド条件入力（贈り先相手、イベント、予算、好みの特徴、自由入力など）は **すべて Request Body に配置する**
- 単一履歴の取得条件である `runId` は **Query ではなく Path Parameter に配置する**
- 履歴一覧の単純な絞り込み条件は **Query Parameter に配置する**

## 11. 単純参照API / 複雑検索API の使い分け

### 11-1. 基本方針

検索ユースケースには、以下の2種類がある。

| 種類        | 用途                                       | HTTP形式    |
| ----------- | ------------------------------------------ | ----------- |
| 単純参照API | 一覧取得、簡易絞り込み、ページング、ソート | GET + Query |
| 複雑検索API | 多条件検索、配列条件、ネスト条件、長文条件 | POST + Body |

### 11-2. 設計原則

- **1つのAPIが2種類の形式を受けるのではなく、用途ごとに別APIとして設計する**
- 外部IFは分ける
- 内部の検索条件モデルは共通化してよい

### 11-3. 例

### 単純参照API

```
GET /api/v1/admin/recommendation-runs?status=success&page=1&pageSize=20
```

### 複雑検索API

```
POST /api/v1/admin/recommendation-runs/search
{
  "statuses": ["success","fail"],
  "createdAt": {
    "from":"2026-04-01T00:00:00Z",
    "to":"2026-04-18T23:59:59Z"
  },
  "recipientRelationships": ["boss","client"],
  "occasions": ["birthday","retirement"],
  "page":1,
  "pageSize":20
}
```

### 11-4. 実装反映方針

- `GET /...` と `POST /.../search` は **別エンドポイント** として定義する
- Request DTO も別にする
- Validation も別にする
- ただしアプリケーション内部では、共通の検索条件モデルへ変換して処理してよい

## 12. HTTPメソッド利用方針

本サービスでは、APIの意味を明確化し、誤用を防ぐため、利用するHTTPメソッドを必要最小限に限定する。

HTTP仕様上利用可能な全メソッドを採用するのではなく、**本アプリケーションの業務意味に合致するもののみを許可する**。

| メソッド | 利用可否 | 用途               | 方針                                                     |
| -------- | -------- | ------------------ | -------------------------------------------------------- |
| GET      | 可       | 参照               | 一覧取得、詳細取得、状態参照に使用する                   |
| POST     | 可       | 実行 / 作成 / 記録 | 実行系API、新規作成、イベント記録、業務操作に使用する    |
| PUT      | 原則禁止 | 全体更新           | 全体置換の業務要件が薄く、誤用リスクが高いため使用しない |
| PATCH    | 原則禁止 | 部分更新           | 更新ルールが曖昧になりやすいため、原則として使用しない   |
| DELETE   | 禁止     | 削除               | 監査性・再現性を重視するため、APIによる削除は提供しない  |

### 12-1. GET利用ルール

- 状態を変更しない参照系処理にのみ使用する
- 一覧取得、詳細取得、参照系管理APIに使用する

### 12-2. POST利用ルール

- 新規作成
- 実行開始
- 行動記録
- 業務意味を持つ状態変更

に使用する

### 12-3. POSTの用途分類

| POST種別 | 意味                 | 例                            |
| -------- | -------------------- | ----------------------------- |
| 作成     | 新しいリソースを作る | `/semantic-configs`           |
| 実行     | 処理を起動する       | `/recommendations`            |
| 操作     | 状態を変える業務操作 | `/{id}/activate`              |
| 複雑検索 | 複雑な検索条件を送る | `/recommendation-runs/search` |

### 12-4. PUT禁止ルール

- 全体置換が必要な業務要件が明確に存在しない限り使用しない

### 12-5. PATCH禁止ルール

- 部分更新APIは原則設けない
- 更新が必要な場合は、業務意味を持つPOSTで表現する

### 12-6. DELETE禁止ルール

- 物理削除APIは提供しない
- 削除相当の要件は、無効化・終了・非表示などの業務状態で表現する

### 12-7. 補足

本サービスでは、RESTの形式的一貫性よりも、**業務意味の明確性・誤用防止・監査性** を優先する。

そのため、API設計は `GET + POST` を基本とし、更新・削除は業務操作として明示的に表現する。

## 13. エラー設計方針

### 13-1. HTTPステータス

| ステータス | 意味         |
| ---------- | ------------ |
| 400        | 入力不正     |
| 401        | 未認証       |
| 403        | 権限不足     |
| 404        | 対象なし     |
| 409        | 状態競合     |
| 422        | 業務エラー   |
| 500        | 内部エラー   |
| 503        | 一時利用不可 |

### 13-2. エラーレスポンス形式

```
{
  "error": {
    "code":"INVALID_ARGUMENT",
    "message":"priceMin must be <= priceMax",
    "details": [],
    "requestId":"xxx"
  }
}
```

### 13-3. 方針

- 利用者向けメッセージと内部原因を分離する
- requestId でトレース可能にする

## 14. 認証・認可方針（IF観点）

本サービスでは、APIの認証・認可は以下の方針で扱う。

・認証はAuthorizationヘッダーによるBearerトークンで行う
・内部APIはサービス間認証を使用する
・認可はAPI単位およびリソース単位で適用する

### API種別ごとの認証要否

| API種別     | 認証 |
| ----------- | ---- |
| 一般API     | 必須 |
| 行動記録API | 必須 |
| 管理API     | 必須 |
| 評価API     | 必須 |
| 内部API     | 必須 |

### エラー表現

・未認証：401
・認可エラー：403
・対象リソースなし：404

### 補足

詳細な認証方式、トークン設計、権限モデルは「認証・認可方針書」を参照すること。

## 18. 最終まとめ

```
APIは軽量な意思決定層に限定し、
URL / Path / Query / Header / Body の責務を明確に分離する。
単純参照は GET + Query、
複雑検索と実行系入力は POST + Body を基本とする。
```

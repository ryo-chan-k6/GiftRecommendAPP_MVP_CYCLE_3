## 1. 目的

本ドキュメントは、本サービスにおける非機能要件を定義することを目的とする。

非機能要件とは、機能要件とは別に、システムの品質・性能・運用・セキュリティ・可用性等を規定する要件であり、以下の設計の前提となる。

- 基盤構成設計
- API設計
- バッチ設計
- ログ・監視設計
- セキュリティ設計

---

## 2. 対象システムの特性

### 2.1 システム概要

- ギフトレコメンドサービス
- ユーザー入力に基づく意味推定＋商品推薦
- 多段階推論（同期処理）
- 行動ログ収集・評価ループあり

---

### 2.2 処理特性

| 種別   | 内容                                 |
| ------ | ------------------------------------ |
| OL処理 | レコメンド実行、行動記録             |
| BT処理 | 商品データ生成、特徴量生成、評価集計 |
| Hybrid | ログ記録（OL）＋分析（BT）           |

---

### 2.3 設計上の重要前提

```
・レコメンド処理は同期処理（UX重視）
・重い処理は可能な限りBTへオフロード
・ログは全記録し後段分析で活用
```

# 3. 非機能要件一覧

## 3.1 性能要件

### 3.1.1 レスポンス性能

| 項目                  | 要件               |
| --------------------- | ------------------ |
| レコメンドAPI応答時間 | 95%タイルで2秒以内 |
| 最大応答時間          | 5秒以内            |
| interaction記録       | 200ms以内          |

---

### 3.1.2 スループット

| 項目           | 要件             |
| -------------- | ---------------- |
| 同時ユーザー数 | MVP: 50〜100     |
| RPS            | MVP: 10〜30      |
| 将来拡張       | 水平スケール可能 |

---

### 3.1.3 ボトルネック対策

| 対策       | 内容                  |
| ---------- | --------------------- |
| キャッシュ | embedding / candidate |
| 非同期化   | 重い推論              |
| 事前計算   | 商品特徴量            |
| 並列化     | retrieval / scoring   |

---

## 3.2 可用性要件

| 項目         | 要件           |
| ------------ | -------------- |

# 4. 非機能設計の原則

```
・重い処理はBTへ逃がす
・ログはすべて残す
・再現性を最優先
・モジュールは疎結合
・外部IFはシンプルに保つ
```

# 6. 一言まとめ

```
UXを損なわない同期レコメンドを実現しつつ、
重い処理は非同期化し、
ログと再現性を軸に改善可能なシステムとする
```

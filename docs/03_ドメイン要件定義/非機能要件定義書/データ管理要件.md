## 1. 目的

本ドキュメントは、本サービスにおけるデータの保存・更新・参照・保持・廃棄・再現性確保に関する要件を定義し、以下を実現することを目的とする。

- レコメンド結果の再現性確保
- 特徴量空間・統計量の一貫性維持
- ログ・評価・改善ループの継続運用
- データ増加時の保守性・運用性確保

---

## 2. スコープ

| 対象             | 内容                                          |
| ---------------- | --------------------------------------------- |
| 業務データ       | recommendation_run, result, interaction, eval |
| モデル関連データ | feature, embedding, μ/σ, version              |
| ログデータ       | run_event, metric, error, audit               |
| バッチ生成データ | 集計・中間・監視用派生データ                  |
| 管理データ       | semantic_config, model_version など           |

---

## 3. データ管理の基本方針

### 3.1 原則

```
・真の状態は永続層に保持する
・再現性を優先する
・append-onlyを基本とする
・更新より追加を優先する
・業務データと観測データを分離する
・高頻度データは集約前提で管理する
```

### 3.2 基本思想

本サービスでは、単なる CRUD データ管理ではなく、**「推薦処理の再現」「品質の説明」「改善の比較」** を成立させることがデータ管理の中心目的となる。

特に、特徴量分布や正規化統計量はレコメンド品質の基盤となるため、通常のアプリケーションデータより強く**版管理・比較可能性・履歴保持**を要求する。

## 4. データ分類

| 分類         | 内容                               | 例                                              |
| ------------ | ---------------------------------- | ----------------------------------------------- |
| 業務データ   | サービス利用そのものに必要なデータ | recommendation_run, recommendation_result       |
| 行動データ   | 利用者行動を表すデータ             | interaction                                     |
| 管理データ   | 設定・バージョン管理データ         | semantic_config, model_version                  |
| モデルデータ | 特徴量・embedding・統計量          | item_feature, user_feature, normalization_stats |
| 観測データ   | 運用・監視用データ                 | metric_log, run_event_log                       |
| 派生データ   | 集約・分析・監視用に生成したデータ | 日次集計、分布統計、品質レポート                |

## 5. 真の状態（Source of Truth）要件

### 5.1 方針

| 対象         | 真の状態                                           |
| ------------ | -------------------------------------------------- |
| 推薦実行履歴 | recommendation_run                                 |
| 推薦結果     | recommendation_result / recommendation_result_item |
| 行動記録     | interaction_log                                    |
| 設定         | semantic_config                                    |
| モデル版     | model_version                                      |
| 正規化統計量 | normalization_stats系                              |
| 特徴量       | item/user feature の正本データ                     |

### 5.2 要件

```
・真の状態はDB等の永続層に保持する
・アプリ内メモリやキャッシュは真の状態とみなさない
・キャッシュは消失しても業務継続できること
```

---

## 6. 再現性要件（最重要）

### 6.1 目的

レコメンド結果・異常・品質劣化を後から説明できるようにする。

### 6.2 必須保持項目

| 項目                         | 理由                       |
| ---------------------------- | -------------------------- |
| runId                        | 実行単位の識別             |
| 入力条件                     | 何を推薦したか再現するため |
| semantic_config_version      | 意味推定条件の再現         |
| model_version                | ランキング条件の再現       |
| 使用feature/統計量バージョン | 空間座標系の再現           |
| result内容                   | 実際に返した結果の再現     |
| fallback有無                 | 品質評価・障害分析のため   |

### 6.3 方針

```
・run単位で再実行可能な情報を保存する
・少なくとも「何を返したか」と「その時の条件」は残す
・結果だけでなく前提バージョンも記録する
```

---

## 7. 版管理要件

### 7.1 管理対象

| 対象                     | 版管理要否 |
| ------------------------ | ---------- |
| semantic_config          | 必須       |
| model_version            | 必須       |
| 正規化統計量             | 必須       |
| item_feature生成ロジック | 将来必要   |
| user_feature生成ロジック | 将来必要   |

### 7.2 要件

```
・版は比較可能であること
・現行版と過去版を区別できること
・run時点でどの版を使ったか追跡できること
```

### 7.3 理由

正規化統計量は「意味空間の座標系そのもの」であり、変更されると比較軸も変わるため、版管理が必須である。

---

## 8. 更新方針

### 8.1 基本方針

| データ種別         | 方針                 |
| ------------------ | -------------------- |
| recommendation_run | append-only          |
| interaction_log    | append-only          |
| run_event_log      | append-only          |
| result系           | append-only          |
| config / version   | 新版追加 + 状態変更  |
| feature / 統計量   | 再生成・差し替え管理 |

### 8.2 原則

```
・履歴が重要なデータは更新ではなく追加する
・物理削除より状態変更・失効を優先する
・再生成可能データはBTで再構築可能にする
```

---

## 9. 保持期間要件

### 9.1 基本方針

保持期間は、**再現性・監査性・コスト**のバランスで定める。

### 9.2 初期方針

| データ                | 保持方針          |
| --------------------- | ----------------- |
| recommendation_run    | 長期保持          |
| recommendation_result | 長期保持          |
| interaction_log       | 長期保持          |
| run_event_log         | 短中期保持 + 集約 |
| metric_log            | 短中期保持 + 集約 |
| error_log             | 中期保持          |
| audit_log             | 中期〜長期保持    |
| 一時派生データ        | 短期保持          |

### 9.3 原則

```
・再現に必要なデータは長期保持
・高頻度ログは集約前提で短中期保持
・原本と集約データの両方を用途に応じて使い分ける
```

---

## 10. 集約・派生データ要件

### 10.1 方針

高頻度データは、そのまま長期間直接参照するのではなく、集約データを生成する。

### 10.2 対象

| 元データ        | 集約単位の例                    |
| --------------- | ------------------------------- |
| interaction_log | 日次CTR、favorite率             |
| metric_log      | 分・時間・日次集計              |
| run_event_log   | ステップ別所要時間集計          |
| feature分布統計 | version別・日次スナップショット |

### 10.3 要件

```
・集約はBTで実施する
・元データと集約データの責務を分離する
・ダッシュボードは原則集約データを利用する
```

---

## 11. モデル・特徴量データ管理要件

### 11.1 feature管理

| 要件     | 内容                     |
| -------- | ------------------------ |
| 管理単位 | feature単位・version単位 |
| 比較     | 過去版比較可能           |
| 再生成   | BTで可能                 |
| 異常検知 | 分布監視対象             |

### 11.2 正規化統計量管理

| 要件             | 内容 |
| ---------------- | ---- |
| μ/σ保持          | 必須 |
| feature単位管理  | 必須 |
| version単位管理  | 必須 |
| 過去版比較       | 必須 |
| 更新時の履歴保持 | 必須 |

### 11.3 設計思想

分布は「理想形に寄せるもの」ではなく「異常を検知するセンサー」であり、統計量は比較・改善の前提となる。したがって、集約値や分布統計は派生情報ではあるが、**改善活動上は重要な管理対象**として扱う。

## 12. ログデータ管理要件

### 12.1 役割分離

| ログ            | 管理目的       |
| --------------- | -------------- |
| run_event_log   | 実行トレース   |
| interaction_log | 行動分析       |
| metric_log      | 指標観測       |
| error_log       | 障害分析       |
| audit_log       | 監査・変更履歴 |

### 12.2 要件

```
・ログは用途別に分離して保持する
・ログ種別ごとに保持期間・集約方針を定義する
・非機能指標に必要な項目はアプリ側で明示的に出力する
```

---

## 13. データ品質要件

### 13.1 観点

| 観点       | 内容                                    |
| ---------- | --------------------------------------- |
| 完全性     | 必須項目が欠落していない                |
| 一貫性     | version / key / relation が整合している |
| 再現性     | 同じ前提で同じ結果を説明できる          |
| 比較可能性 | 過去との比較ができる                    |
| 鮮度       | 期待する更新頻度を満たす                |

### 13.2 品質異常例

| 異常                           | 内容         |
| ------------------------------ | ------------ |
| runにversionが記録されていない | 再現不能     |
| μ/σ更新履歴がない              | 比較不能     |
| interactionがrunに紐づかない   | 行動分析不能 |
| feature統計の母集団定義が不明  | 異常検知不能 |

---

## 14. 削除・廃棄要件

### 14.1 基本方針

```
・物理削除は最小限
・原則は失効・状態変更・保持期限切れで制御する
```

### 14.2 理由

- recommendation_run や interaction は履歴価値が高い
- 監査・分析・改善の観点で物理削除は不利
- MVP段階では削除より保持設計を優先する

### 14.3 例外

- 一時データ
- 再生成可能な中間データ
- 保持期限を超過した高頻度メトリクス原本

## 15. バックアップ要件

### 15.1 対象

| 対象                      | バックアップ要否 |
| ------------------------- | ---------------- |
| recommendation_run/result | 必須             |
| interaction_log           | 必須             |
| config/version            | 必須             |
| 正規化統計量              | 必須             |
| audit_log                 | 必須             |
| 一時派生データ            | 任意             |

### 15.2 方針

```
・重要データは定期バックアップする
・復旧時に最低限、run / result / config/version を戻せること
```

---

## 16. セキュリティ・プライバシー要件

### 16.1 方針

- 個人情報は最小限保持
- ログに不要な自由入力全文を残さない
- JWT全文や秘密情報は保存しない
- 分析用データも必要最小限の識別情報に留める

### 16.2 具体要件

| 項目                     | 要件                     |
| ------------------------ | ------------------------ |
| token全文保存            | 禁止                     |
| パスワード保存           | 禁止                     |
| 個人情報過剰保持         | 禁止                     |
| 自由入力全文の無制限保持 | 原則禁止またはマスキング |

---

## 17. スケール対応要件

### 17.1 対象

| データ          | 課題        | 対応                          |
| --------------- | ----------- | ----------------------------- |
| interaction_log | 増加量大    | 集約・partition               |
| metric_log      | 高頻度      | 短期保持＋集約                |
| run_event_log   | 高頻度      | ステップ単位集約              |
| feature統計     | version増加 | version管理＋スナップショット |

### 17.2 方針

```
・高頻度データは集約前提で管理する
・参照用途に応じて原本と集約を分ける
・将来のpartition / archiveを見越した構造にする
```

---

## 18. 運用要件との接続

データ管理は単なる保存ではなく、以下の運用活動を支える必要がある。

- 障害時の再現
- 品質低下の原因分析
- FB分類との連携
- 分布異常検知
- A/B比較
- 改善前後比較

フィードバック分類では、User Modeling / Item / Retrieval / Ranking / 制約整合 / 分布・空間の各レイヤに壊れ方をラベル付けして改善に接続する前提となっており、データ管理側でもこれを支える粒度が必要である。

## 20. 設計原則まとめ

```
・真の状態は永続層に置く
・再現性を最優先する
・履歴は追加中心で管理する
・高頻度データは集約前提とする
・特徴量・統計量は版管理と比較可能性を必須とする
・データ管理は改善ループを支える前提で設計する
```

```
本サービスのデータ管理は、
「保存すること」ではなく

を目的として設計する
```

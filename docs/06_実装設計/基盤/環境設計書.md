## 1. 目的

本ドキュメントは、本サービスにおける実行環境の構成方針を定義し、以下を明確化することを目的とする。

- 開発・検証・本番の環境分離方針
- 各環境に配置するコンポーネント
- 環境ごとの役割と品質水準
- データ・設定・認証情報の分離方針
- 後続の DevOps / CI/CD / 運用設計の前提

---

## 2. スコープ

| 対象     | 内容                           |
| -------- | ------------------------------ |
| 実行環境 | dev / stg / prod               |
| アプリ   | Web / API / Reco / Batch       |
| データ   | PostgreSQL / Redis             |

| 監視     | ログ / メトリクス / アラート   |
| 外部依存 | Embedding API / 外部商品データ |

## 3. 環境設計の基本方針

### 3.1 基本方針

```
・環境ごとに役割を明確に分ける
・本番と非本番は必ず分離する
・環境差分は最小化しつつ、品質ゲートは段階的に高める
・Secrets / DB / Cache は環境ごとに分離する
・アプリ設定はコード埋め込みではなく環境設定で切り替える
```

### 3.2 役割の原則

| 環境 | 主目的                   |
| ---- | ------------------------ |
| dev  | 実装・動作確認           |
| stg  | リリース前検証・結合確認 |
| prod | 実利用サービス提供       |

---

## 4. 環境一覧

## 4.1 環境定義

| 環境             | 論理名 | 目的                               |
| ---------------- | ------ | ---------------------------------- |
| 開発環境         | dev    | 実装・単体確認・疎通確認           |
| ステージング環境 | stg    | 本番前検証・結合確認・簡易性能確認 |
| 本番環境         | prod   | ユーザー提供用                     |

---

## 4.2 MVP方針

### 事実

MVP段階では、環境を増やしすぎると運用コストが上がる。

### 推論

そのため MVP では、

```
dev / prod を先行整備し、
stg は必要に応じて導入、または簡易構成で開始する
```

でもよい。

ただし、**本番前検証の場は必ず確保する**。

## 5. 環境別の役割

## 5.1 dev

### 目的

- 実装
- 単体確認
- ローカル疎通
- API / Reco / Batch の基本動作確認

### 特徴

| 項目       | 方針           |
| ---------- | -------------- |
| 安定性     | 高くなくてよい |
| データ品質 | 簡易データで可 |
| 監視       | 最小限         |
| 可用性     | 要求しない     |
| 再現性     | ある程度で可   |

---

## 5.2 stg

### 目的

- 本番相当構成での結合確認
- リリース前確認
- 画面/API/Batch の整合確認
- 主要シナリオの通し確認

### 特徴

| 項目       | 方針                     |
| ---------- | ------------------------ |
| 安定性     | 本番に近づける           |
| データ品質 | 本番準拠のサンプルデータ |
| 監視       | 本番に近い               |
| 可用性     | 中程度                   |
| 再現性     | 必須                     |

---

## 5.3 prod

### 目的

- 実利用提供
- 品質監視
- 改善ループ運用

### 特徴

| 項目       | 方針           |
| ---------- | -------------- |
| 安定性     | 最重視         |
| データ品質 | 正式データのみ |
| 監視       | 必須           |
| 可用性     | 最重視         |
| 再現性     | 必須           |

---

## 7. 環境別データ方針

## 7.1 DB分離

```
・環境ごとにDBを分離する
・prodデータをdev/stgで直接共有しない
```

### 理由

- 誤更新防止
- 検証データ混入防止
- セキュリティ確保

---

## 7.2 Redis分離

```
・環境ごとにRedisを分離する
・prod cache を dev/stg と共有しない
```

---

## 7.3 データ投入方針

| 環境 | 方針                             |
| ---- | -------------------------------- |
| dev  | サンプルデータ / 小規模データ    |
| stg  | 本番準拠データ（匿名化・縮小版） |
| prod | 正式データ                       |

---

## 8. 環境別設定管理方針

## 8.1 基本方針

```
・設定は環境変数またはSecretsで切り替える
・コード内に環境依存値を埋め込まない
```

---

## 8.2 環境ごとに分けるべきもの

| 項目               | 分離要否 |
| ------------------ | -------- |
| DB接続先           | 必須     |
| Redis接続先        | 必須     |
| API Base URL       | 必須     |
| Reco Base URL      | 必須     |
| Embedding API Key  | 必須     |
| JWT秘密情報        | 必須     |
| ログレベル         | 推奨     |
| バッチスケジュール | 推奨     |

---

## 8.3 Secrets管理方針

```
・Secretsは環境ごとに別管理
・共通Secretsを使い回さない
・本番Secretsは開発環境で利用しない
```

## 10. 環境別監視方針

## 10.1 dev

- ローカルログ中心
- 重大エラーのみ確認
- ダッシュボードは簡易でも可

## 10.2 stg

- API / Reco の latency / error を確認
- Batch成功可否を確認
- 主要メトリクスをダッシュボード表示

## 10.3 prod

- System / App / Business / Model の4層監視を実施
- アラート運用を行う
- runId を軸に追跡可能とする

## 17. 設計原則まとめ

```
・環境ごとに役割を分ける
・本番と非本番は必ず分離する
・設定差分は環境変数で吸収する
・品質要求は dev < stg < prod の順に高める
・MVPでは最小構成で開始し、必要に応じて stg を強化する
```

---

## 18. 一言まとめ

```
本サービスの環境設計は、
「開発効率」と「本番安全性」を両立するために、
dev / stg / prod を役割ごとに分離し、
構成差分を最小限に保ちながら品質ゲートを段階化する
```

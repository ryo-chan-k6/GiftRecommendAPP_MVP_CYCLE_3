## 1. 目的

本設計書は、Gift Recommendation Service の開発を継続可能かつ高効率に進めるために、以下を定義する。

- リポジトリ構造
- コード配置ルール
- 設計書管理方針（docs正本）
- AI開発前提の構成
- 責務分離と依存関係制御

目的は単なるフォルダ設計ではなく、以下を成立させること。

- Cursor中心で開発が完結する
- 設計と実装が常に一致する
- Issue / Branch / CI/CD に自然接続できる
- 将来チーム開発でも破綻しない

## 5.5 運用ルール

- 設計変更 = docs更新必須
- コードと同一PRで管理
- ADRで意思決定記録
- docs運用ルールは、`docs/README.md`に記載する。
  README.mdの内容は少なくとも下記の内容を含めることとする。

```
# docs 運用ルール

## 正本管理
- 本ディレクトリを設計書の正本とする
- Notionは副本とする

## 更新ルール
- 設計変更を伴うPRでは docs 更新を必須とする
- コード変更と同一PRで管理する

## 命名ルール
- フォルダ：工程ベース
- ファイル：成果物名に準拠

## 禁止事項
- docsを更新せずに仕様変更すること
- Notionのみ更新すること
```

- また、PRテンプレにdocsチェックを追加（`.github/pull_request_template.md`）することでdocs正本運用を強制する。pull_request_template.mdの内容サンプルは下記とする。
  ※docsは開発運用上、絶対の正本とする方針。docsの修正は漏れがちなので、きちんとケアする。

```
## 変更内容

## 関連Issue

## docs更新
- [ ] docs更新あり
- [ ] docs更新不要（理由記載）

## 確認内容
```

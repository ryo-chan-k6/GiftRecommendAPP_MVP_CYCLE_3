## 1. 目的

本ドキュメントは、本サービスにおける保守性および拡張性の要件を定義し、

- 安全に変更できること
- 影響範囲を限定できること
- 機能追加・改善を継続できること

を実現することを目的とする。

---

## 2. 基本思想（最重要）

```
・変更は前提である
・壊さずに変えられる設計にする
・責務単位で分離する
・データとロジックを分離する
・観測と改善を前提にする
```

---

## 3. 保守性と拡張性の定義

| 観点   | 定義                                 |
| ------ | ------------------------------------ |
| 保守性 | 既存機能を安全に修正・改善できること |
| 拡張性 | 新機能や新ロジックを追加できること   |

---

# 4. アーキテクチャ要件

---

## 4.1 サービス分離

```
API（I/F層）
Reco（ドメインロジック）
```

---

### 要件

| 項目 | 内容         |
| ---- | ------------ |
| API  | 入出力・制御 |
| Reco | 推論ロジック |
| 分離 | 必須         |

---

### 理由

```
・Recoロジックの変更頻度が高い
・APIは安定させたい
```

---

---

## 4.2 モジュール分割

---

### 原則

```
1モジュール = 1責務
```

---

### 対象

| モジュール         | 内容       |
| ------------------ | ---------- |
| input_parse        | 入力解析   |
| meaning_estimation | 意図推定   |
| retrieval          | 候補抽出   |
| ranking            | スコア計算 |
| output             | 出力整形   |

---

### 要件

```
・各モジュールは独立変更可能
・内部依存を最小化
```

---

---

## 4.3 疎結合設計

---

### 要件

```
・モジュール間はインターフェース経由
・内部実装を隠蔽
```

---

### NG

```
・直接DBアクセス共有
・内部構造依存
```

---

---

# 5. データ分離要件

---

## 5.1 データとロジック分離

---

### 対象

| 分離対象        | 内容       |
| --------------- | ---------- |
| semantic_config | ルール     |
| model_version   | パラメータ |
| feature         | 入力       |
| code            | 処理       |

---

### 要件

```
・ロジックはコード
・パラメータはデータ
```

---

---

## 5.2 設定駆動設計

---

```
コード変更なしで挙動変更可能
```

---

例：

- λ調整
- feature重み
- ルール定義

## 1. 目的

本ドキュメントは、本サービスにおける情報セキュリティ要件を定義し、

認証・認可・通信・データ保護・脆弱性対策を通じて、安全なシステムを構築するための指針とする。

---

## 2. 対象範囲

| 対象         | 内容           |
| ------------ | -------------- |
| クライアント | ブラウザ       |
| Webサーバ    | フロントエンド |
| API          | 外部公開API    |
| Reco         | 内部サービス   |
| DB           | データストア   |
| Cache        | Redis等        |
| インフラ     | デプロイ基盤   |

---

## 3. セキュリティ設計原則

```
・信頼境界を明確にする
・最小権限の原則
・ゼロトラスト前提
・入力はすべて不正とみなす
・ログは追跡可能にする
```

---

## 4. 認証要件（Authentication）

---

### 4.1 認証方式

| 項目         | 内容                       |
| ------------ | -------------------------- |
| 方式         | JWT（Bearer Token）        |
| セッション   | 使用しない（ステートレス） |
| トークン送信 | Authorizationヘッダ        |

---

### 4.2 トークン仕様

| 項目         | 内容         |
| ------------ | ------------ |
| 有効期限     | 1時間（MVP） |
| リフレッシュ | 将来対応     |
| 署名         | 必須         |
| 改ざん検知   | 必須         |

---

### 4.3 トークン検証

```
・署名検証
・有効期限チェック
・issuer/audienceチェック
```

---

### 4.4 認証対象

| API     | 認証                   |
| ------- | ---------------------- |
| 一般API | 必須                   |
| 管理API | 必須（強化）           |
| 内部API | 必須（サービス間認証） |

---

## 5. 認可要件（Authorization）

---

### 5.1 ロール設計

| ロール    | 権限             |
| --------- | ---------------- |
| user      | 自分のデータのみ |
| admin     | 全データ         |
| evaluator | 評価データ       |

---

### 5.2 認可制御

```
・リソース単位で制御
・userは自分のrunのみ参照可能
・adminは全件アクセス可能
```

---

### 5.3 実装

| レイヤ | 内容                 |
| ------ | -------------------- |
| API    | 認可チェック実施     |
| Reco   | 認可不要（内部処理） |

---

## 6. 通信セキュリティ

---

### 6.1 HTTPS

| 項目       | 要件      |
| ---------- | --------- |
| プロトコル | HTTPS必須 |
| HTTP       | 禁止      |
| TLS        | 最新推奨  |

---

### 6.2 内部通信

```
・API ⇄ Reco もHTTPS
・内部ネットワークでも暗号化推奨
```

---

## 7. 入力検証（Validation）

---

### 7.1 要件

```
・全入力を検証
・型チェック
・長さ制限
・許可値制限
```

---

### 7.2 対象

| 対象     | 内容 |
| -------- | ---- |
| API Body | 必須 |
| Query    | 必須 |
| Path     | 必須 |

---

### 7.3 NG対策

| 攻撃          | 対策                 |
| ------------- | -------------------- |
| SQL Injection | ORM / プレースホルダ |
| XSS           | エスケープ           |
| コマンド注入  | 禁止                 |

---

## 8. データ保護

---

### 8.1 個人情報

```
・最小限のみ保持
・ログに含めない
```

---

### 8.2 機密情報

| 項目       | 管理        |
| ---------- | ----------- |
| APIキー    | 環境変数    |
| DB接続情報 | Secrets管理 |
| JWT秘密鍵  | 厳重管理    |

---

### 8.3 保存データ

```
・平文パスワード禁止
・必要に応じて暗号化
```

---

## 9. ログセキュリティ

---

### 9.1 禁止事項

```
・パスワードログ出力禁止
・トークン出力禁止
・個人情報出力禁止
```

---

### 9.2 必須ログ

| 項目     | 内容 |
| -------- | ---- |
| 認証失敗 | 記録 |
| 認可違反 | 記録 |
| 重要操作 | 記録 |

---

## 10. APIセキュリティ

---

### 10.1 レート制限

| 項目 | 内容      |
| ---- | --------- |
| 対象 | 全API     |
| 制限 | MVPは緩め |
| 将来 | 必須      |

---

### 10.2 CORS

```
・許可ドメイン限定
```

---

### 10.3 CSRF

```
・JWT使用で基本回避
・Cookie使用時は対策必須
```

---

## 11. サービス間認証

---

### 11.1 API ⇄ Reco

```
・内部トークン使用
・IP制限（将来）
```

---

## 12. 脆弱性対策

---

### 12.1 対象

| 攻撃 | 対策       |
| ---- | ---------- |
| XSS  | エスケープ |
| CSRF | トークン   |
| SQLi | ORM        |
| DoS  | レート制限 |

---

### 12.2 ライブラリ

```
・定期アップデート
・脆弱性スキャン
```

---

## 13. 権限分離

---

### 13.1 環境分離

| 環境 | 分離       |
| ---- | ---------- |
| dev  | 本番と分離 |
| prod | 厳格管理   |

---

### 13.2 アクセス制御

```
・管理APIは限定公開
```

---

## 14. セキュリティログ

---

### 14.1 記録内容

```
・認証失敗
・不正アクセス
・権限違反
```

---

### 14.2 保持

| 項目     | 内容     |
| -------- | -------- |
| 保存期間 | 7日以上  |
| 監査     | 将来対応 |

---

## 15. セキュリティ設計原則まとめ

```
・ステートレス認証
・最小権限
・入力検証徹底
・ログで追跡可能
・秘密情報は外部管理
```

---

## 16. 一言まとめ

```
ステートレスなJWT認証を前提に、
認証・認可・通信・入力検証・データ保護を徹底し、
安全かつ拡張可能なシステムを構築する
```

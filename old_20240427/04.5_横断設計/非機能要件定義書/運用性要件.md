# 運用性要件（Operations Requirements）

---

## 1. 目的

本ドキュメントは、本サービスの運用に関する要件を定義し、

- 障害対応
- 性能維持
- レコメンド品質維持・改善

を継続的に実行可能にすることを目的とする。

---

## 2. 運用の全体構造

```mermaid
flowchart LR
    A[観測] --> B[検知]
    B --> C[判断]
    C --> D[対応]
    D --> E[改善]
```

---

## 3. 運用対象の分類

| 分類         | 内容                 |
| ------------ | -------------------- |
| システム運用 | インフラ・API・Reco  |
| データ運用   | DB・ログ・バッチ     |
| モデル運用   | 特徴量・分布・スコア |
| 業務運用     | 評価・改善           |

---

# 🔴 ここがコア（あなたのサービスの本質）

---

## 4. モデル運用要件（最重要）

---

## 4.1 目的

```
レコメンド品質の劣化を検知し、
原因特定と改善を継続的に回す
```

---

## 4.2 運用サイクル

```
分布監視
↓
異常検知
↓
FB分類
↓
修正箇所特定
↓
改善
```

---

## 4.3 運用対象

| 対象        | 内容                        |
| ----------- | --------------------------- |
| feature分布 | μ / σ / 分布                |
| 空間構造    | social / symbolic           |
| スコア構造  | context / popularity / risk |
| interaction | user × item                 |
| eval結果    | human評価                   |

---

## 4.4 異常検知運用

---

### 検知対象

| 異常           | 内容           |
| -------------- | -------------- |
| 分布崩壊       | σ低下、偏り    |
| 空間崩壊       | 1軸化          |
| スコア偏り     | popularity偏重 |
| マッチング異常 | 類似度異常     |

---

### 判断基準

```
・統計的異常（閾値）
・過去比較
・評価結果悪化
```

---

---

## 4.5 フィードバック運用

---

### 方針

```
全評価は必ずFB分類に変換する
```

---

### 処理

```
評価
↓
FB分類
↓
原因レイヤ特定
↓
修正対象決定
```

---

---

## 4.6 改善対象の分類

| レイヤ    | 修正対象      |
| --------- | ------------- |
| user      | 意図推定      |
| item      | feature       |
| retrieval | 候補          |
| ranking   | スコア        |
| 空間      | 分布 / 正規化 |

---

---

## 4.7 改善実行

| 手段        | 内容        |
| ----------- | ----------- |
| ルール修正  | mapping改善 |
| feature修正 | 再推定      |
| 重み調整    | λ調整       |
| 正規化更新  | μ/σ更新     |

---

---

## 4.8 改善検証

```
変更前後で比較
・分布
・スコア
・評価結果
```

# 5. システム運用要件

---

## 5.1 障害対応

| 項目     | 要件      |
| -------- | --------- |
| 検知     | 1分以内   |
| 通知     | 即時      |
| 初動対応 | 10分以内  |
| 復旧     | 1時間以内 |

```
↓
再発防止
```

---

## 5.3 フォールバック運用

```

・閾値超過で調査
```

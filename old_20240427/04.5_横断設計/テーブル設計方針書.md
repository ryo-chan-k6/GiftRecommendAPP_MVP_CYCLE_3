## 1. 目的

本ドキュメントは、本サービスにおけるテーブル設計の共通方針を定義し、後続の

- テーブル一覧
- 物理ER図
- DDL
- インデックス設計
- パーティション設計

に直接適用できるルールを明確化することを目的とする。

---

## 2. スコープ

本方針書は以下の永続化対象に適用する。

| 対象                | 内容                                         |
| ------------------- | -------------------------------------------- |
| 業務テーブル        | recommendation_run, result, interaction など |
| 管理テーブル        | semantic_config, model_version など          |
| ログテーブル        | run_event_log, metric_log, audit_log など    |
| 派生テーブル        | 集約、監視、統計量、特徴量など               |
| version管理テーブル | 統計量、設定、モデル版など                   |

対象外は以下。

| 対象外                       | 理由                     |
| ---------------------------- | ------------------------ |
| 一時メモリキャッシュ         | テーブル管理対象外       |
| Redis等のキャッシュ保存      | 別途キャッシュ戦略で管理 |
| 外部SaaSの監視メトリクス保存 | 基盤側で管理             |

---

## 3. テーブル設計の基本原則

### 3.1 原則

```
・テーブルは業務上の責務単位で分割する
・正本と派生を分離する
・履歴価値の高いデータは append-only を基本とする
・頻繁に変わるロジックは version として管理する
・物理削除より状態管理・失効管理を優先する
・再現性を優先し、run時点の前提を保持する
```

### 3.2 設計思想

本サービスでは、単なるCRUDアプリではなく、

- 推薦結果の再現
- モデル改善前後の比較
- 分布異常・品質劣化の説明
- フィードバックループ

が重要である。

そのため、テーブル設計は「最新状態だけ持てばよい」ではなく、**履歴・版・比較可能性を前提**にする。

## 4. テーブル命名規則

## 4.1 基本ルール

| 項目     | 方針                     |
| -------- | ------------------------ |
| 命名形式 | snake_case               |
| 名称     | 単数形を基本             |
| 接頭辞   | 原則不要                 |
| 接尾辞   | 意味が明確な場合のみ使用 |
| 省略     | 原則しない               |

### 4.2 命名の考え方

```
テーブル名 = 業務リソース名 または 永続化対象概念名
```

例：

- recommendation_run
- recommendation_result
- recommendation_result_item
- interaction_log
- run_event_log
- semantic_config
- model_version

### 4.3 単数形を基本とする理由

DDL・ER・FKカラム名との整合性を優先するため、テーブル名は単数形を基本とする。

例：

- `recommendation_run.run_id`
- `recommendation_result.run_id`

複数形にすると、英語上の揺れや FK命名とのズレが起きやすいため採用しない。

### 4.4 テーブル名の禁止事項

| 禁止                                 | 理由                     |
| ------------------------------------ | ------------------------ |
| 業務意味のない略語                   | 可読性低下               |
| module_a_result のような実装都合命名 | 責務が不明瞭             |
| tmp*, work* の常設化                 | 一時用途が正本化しやすい |

---

## 5. カラム命名規則

## 5.1 基本ルール

| 項目     | 方針                               |
| -------- | ---------------------------------- |
| 命名形式 | snake_case                         |
| 主キー   | `{table_name}_id` または意味単位id |
| 外部キー | 参照先概念名 + `_id`               |
| 真偽値   | `is_` / `has_`                     |
| 日時     | `_at`                              |
| 日付     | `_date`                            |
| 件数     | `_count`                           |
| 率・割合 | `_rate`                            |
| 版       | `_version_id` or `_version_no`     |

### 5.2 ID命名

原則として、主キー名は以下のどちらかに統一する。

| パターン           | 例                         |
| ------------------ | -------------------------- |
| 概念名\_id         | recommendation_run_id      |
| 業務慣用短縮名\_id | run_id, result_id, task_id |

本サービスでは、可読性と実務上の扱いやすさから、**ドメインで定着している短縮形がある場合は短縮形を許可**する。

推奨例：

- run_id
- result_id
- task_id
- config_id
- model_version_id

### 5.3 共通監査カラム

| カラム     | 用途             |
| ---------- | ---------------- |
| created_at | 作成日時         |
| updated_at | 更新日時         |
| created_by | 作成者（必要時） |
| updated_by | 更新者（必要時） |

append-only テーブルでは `updated_at` を持たない、または運用しないことを許容する。

## 6. 主キー方針

## 6.1 基本方針

```
・全テーブルに主キーを持たせる
・業務キーが存在しても、内部管理用の主キーを基本とする
・単一主キーを原則とし、複合主キーは原則避ける
```

### 6.2 主キーの型

| 項目     | 方針                               |
| -------- | ---------------------------------- |
| 基本     | UUID または文字列ID                |
| 採番連番 | 原則不採用                         |
| 理由     | 外部露出・分散生成・将来拡張に強い |

### 6.3 例外

集約テーブルや中間派生テーブルでは、複合ユニーク制約を設けることは可とする。

ただし、**主キーそのものは単一IDを持つことを原則**とする。

---

## 7. 外部キー方針

## 7.1 基本方針

```
・論理的な参照関係は明示する
・物理FK制約は、整合性メリットと運用コストを比較して採用する
・高頻度BT/ログ系は物理FKを緩めることを許容する
```

### 7.2 物理FKの採用基準

| テーブル種別       | 方針                         |
| ------------------ | ---------------------------- |
| 業務正本テーブル   | 原則採用                     |
| 管理テーブル       | 原則採用                     |
| ログテーブル       | 原則は論理参照、物理FKは慎重 |
| 高頻度派生テーブル | 論理参照中心                 |

### 7.3 理由

### 事実

- FK制約は整合性を高める
- 一方で、バッチ投入順序・大量INSERT・ログ高頻度書き込みではボトルネックになりうる

### 方針

- **業務の正本・管理対象はFKで守る**
- **ログ・集計・派生は論理整合＋アプリ保証を許容する**

### 7.4 ON DELETE / ON UPDATE

物理削除を原則行わないため、`ON DELETE CASCADE` は原則不採用。

必要に応じて `RESTRICT` または論理削除で運用する。

---

## 8. 業務キー・ユニーク制約方針

## 8.1 基本方針

```
・主キーとは別に、業務上重複してはならないキーにはユニーク制約を付与する
```

### 8.2 例

| 対象                    | 例                                |
| ----------------------- | --------------------------------- |
| semantic_config version | (config_group, version_no)        |
| model_version           | (model_family, version_no)        |
| 集約テーブル            | (aggregation_date, dimension_key) |

### 8.3 注意

自然キーが存在しても、主キーの代替にはしない。

自然キーはユニーク制約として扱う。

---

## 9. 履歴管理方針

## 9.1 基本方針

```
・履歴価値の高いデータは append-only とする
・更新ではなく追加で履歴を残す
・状態変化は状態遷移として表現する
```

### 9.2 append-only 対象

| テーブル                   | 方針        |
| -------------------------- | ----------- |
| recommendation_run         | append-only |
| recommendation_result      | append-only |
| recommendation_result_item | append-only |
| interaction_log            | append-only |
| run_event_log              | append-only |
| metric_log                 | append-only |
| error_log                  | append-only |
| audit_log                  | append-only |
| human_eval_result          | append-only |

### 9.3 状態管理対象

更新を持つ場合でも、履歴を消さずに状態遷移で扱う。

例：

- semantic_config
- model_version
- human_eval_task

### 9.4 論理削除

論理削除が必要な場合は、`is_deleted` よりも **status管理** を優先する。

理由は、業務意味が明確で履歴説明性が高いため。

---

## 10. version管理方針

## 10.1 基本方針

```
・ロジック変更可能なものは version で管理する
・run時点で使用した version を必ず記録する
・現行版と履歴版を区別できること
```

### 10.2 version管理対象

| 対象              | 管理方法        |
| ----------------- | --------------- |
| semantic_config   | version管理必須 |
| model_version     | version管理必須 |
| 正規化統計量      | version管理必須 |
| feature生成定義   | 将来対象        |
| scoring parameter | version管理推奨 |

### 10.3 テーブル設計パターン

### パターンA: 親 + version テーブル

例：

- semantic_config
- semantic_config_version

### パターンB: 単一テーブル内に version_no を持つ

小規模管理対象に限り許容

### 10.4 推奨

本サービスでは、履歴比較・現行版管理・切替の明確化のため、**親テーブル + versionテーブル構成を推奨**する。

## 11. 正本 / 派生データの区分方針

## 11.1 定義

| 区分       | 定義                                       |
| ---------- | ------------------------------------------ |
| 正本データ | 業務上の真の状態を表すデータ               |
| 派生データ | 正本から計算・集約・変換して生成するデータ |

### 11.2 基本方針

```
・正本と派生は別テーブルに分離する
・派生は再生成可能であることを前提とする
・正本を派生で上書きしない
```

### 11.3 正本の例

- recommendation_run
- recommendation_result
- recommendation_result_item
- interaction_log
- semantic_config
- model_version
- normalization_stats

### 11.4 派生の例

- 日次集約テーブル
- feature分布統計テーブル
- ranking監視集計テーブル
- dashboard用集約テーブル

### 11.5 注意

「表示上便利だから」という理由で派生カラムを正本テーブルへ混在させない。

必要なら materialized view または派生テーブルで表現する。

---

## 12. ログテーブル設計方針

## 12.1 基本方針

```
・ログは用途別に分離する
・高頻度書き込みを前提に簡潔な構造にする
・業務正本とは責務を分離する
```

### 12.2 ログ種別

| テーブル        | 用途             |
| --------------- | ---------------- |
| run_event_log   | 推薦処理トレース |
| interaction_log | ユーザー行動     |
| metric_log      | 観測指標         |
| error_log       | 例外・障害       |
| audit_log       | 重要変更監査     |

### 12.3 ログ設計ルール

| 項目         | 方針                      |
| ------------ | ------------------------- |
| 書き込み     | append-only               |
| 更新         | 原則なし                  |
| FK           | 論理参照中心              |
| インデックス | 時間 + 追跡キー中心       |
| 追跡キー     | request_id, run_id 等必須 |

### 12.4 JSONカラムの扱い

ログでは柔軟性確保のため JSON カラムを許容する。

ただし、**検索・集計で多用する項目は正規カラム化**する。

---

## 13. パーティション方針

## 13.1 基本方針

```
・高頻度かつ時系列性の高いテーブルは、将来のパーティション対象とする
・MVPでは過剰設計を避けるが、後から分割しやすいキー設計にする
```

### 13.2 対象候補

| テーブル        | パーティション候補 |
| --------------- | ------------------ |
| interaction_log | created_at         |
| run_event_log   | created_at         |
| metric_log      | created_at         |
| error_log       | created_at         |
| audit_log       | created_at         |

### 13.3 パーティションキー

原則 `created_at` ベースの時系列分割を推奨する。

### 13.4 MVP方針

MVPでは即時導入は必須ではない。

ただし、**時系列カラムを必須とし、後から月次・日次分割できる構造**にしておく。

---

## 14. 集約方針

## 14.1 基本方針

```
・高頻度ログは生データと集約データを分ける
・ダッシュボード用途は集約テーブルを使う
・集約はBTで実施する
```

### 14.2 集約対象

| 元データ        | 集約単位       |
| --------------- | -------------- |
| interaction_log | 日次・週次     |
| metric_log      | 分・時・日     |
| run_event_log   | ステップ別日次 |
| feature統計     | version別日次  |

### 14.3 集約テーブル命名

集約テーブルは以下を推奨する。

```
{base_table}_daily_summary
{base_table}_hourly_summary
```

例：

- interaction_log_daily_summary
- metric_log_hourly_summary

---

## 15. インデックス設計方針

## 15.1 基本方針

```
・主キー以外に、参照パターンに応じたインデックスを付与する
・先にアクセスパターンを定義し、やみくもに貼らない
```

### 15.2 優先インデックス

| パターン     | 例                   |
| ------------ | -------------------- |
| ID参照       | run_id               |
| 時系列参照   | created_at           |
| 所有者参照   | user_id + created_at |
| 管理一覧参照 | status + created_at  |
| ログ追跡     | run_id + created_at  |

### 15.3 禁止事項

- 将来使うかも、だけで貼る
- JSON全文検索前提の無秩序な保存
- 重複インデックス乱立

---

## 16. nullable方針

## 16.1 基本方針

```
・業務上必須の項目は NOT NULL
・将来未定義だからという理由で安易に NULL を許可しない
```

### 16.2 NULL許容基準

| 許容する例             | 理由       |
| ---------------------- | ---------- |
| 将来値が入る監査項目   | 任意       |
| fallback時のみ入る項目 | 条件付き   |
| optional入力           | 業務上任意 |

---

## 17. ステータス管理方針

## 17.1 基本方針

```
・状態を持つ概念は status カラムで管理する
・状態遷移設計書と整合すること
```

### 17.2 対象例

- recommendation_run
- human_eval_task
- semantic_config_version
- model_version

### 17.3 注意

status は単なる文字列ではなく、**遷移可能性が定義された業務状態**として扱う。

---

## 18. 日時管理方針

## 18.1 基本方針

| 項目     | 方針                                         |
| -------- | -------------------------------------------- |
| 保存形式 | timestamptz 相当                             |
| 業務時刻 | created_at / updated_at / occurred_at を区別 |
| 集計日   | dateカラムを別途持つことを許容               |

### 18.2 時刻の意味分離

| カラム        | 意味             |
| ------------- | ---------------- |
| created_at    | レコード作成時刻 |
| updated_at    | 更新時刻         |
| occurred_at   | イベント発生時刻 |
| aggregated_at | 集約生成時刻     |

---

## 19. DDL作成時の必須確認項目

DDL着手前に、各テーブルについて以下を必ず確定する。

| 項目         | 内容                      |
| ------------ | ------------------------- |
| テーブル種別 | 正本 / 派生 / ログ / 管理 |
| 主キー       | 何を採用するか            |
| 業務キー     | 何をユニークとするか      |
| FK           | 物理制約を張るか          |
| 更新方針     | append-only / update可    |
| status有無   | 状態管理対象か            |
| version有無  | 履歴比較対象か            |
| created_at   | 時系列管理対象か          |
| 集約対象か   | summary必要か             |
| 保持期間     | 長期 / 短中期             |

## 20. MVP方針

## 20.1 MVPで必須

- 命名規則統一
- 主キー / FK方針統一
- append-only の徹底
- version管理対象の明確化
- 正本 / 派生分離
- ログテーブル分離
- 時系列カラム整備

## 20.2 MVPでは後回し可

- 物理パーティション即時導入
- 高度なアーカイブ自動化
- すべての派生集約テーブルの先行作成

## 21. 設計原則まとめ

```
・テーブルは責務単位で分ける
・履歴価値の高いものは append-only
・version と status で変更可能性を管理する
・正本と派生を混ぜない
・ログは用途別に分離する
・時系列増加を前提に、後から集約・分割できる構造にする
```

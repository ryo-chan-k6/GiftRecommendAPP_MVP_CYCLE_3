# 1. このフェーズでの前提

## 事実

Phase1 の目的は、以前整理した通りです。

- 意味ベースで推薦が成立する状態を作る
- 明らかに不適切な推薦を避ける
- ロジックの説明ができる状態にする

## 推論

したがって Phase1 では、機械学習の精度指標を厳密に追うよりも、まずは次を重視すべきです。

- **推薦結果が人間感覚として破綻していないか**
- **心理モデルが最低限動いているか**
- **後で改善できるだけの観測情報が取れるか**

つまり、Phase1 の検証は

**「正答率を測る」より「破綻検知と仮説妥当性確認」** が中心です。

| P1-V2      | 商品 meaning が妥当に付与されるか           | Item Meaning Extraction の妥当性 |

# 3. Phase1 の検証方針

## 結論

Phase1 は、次の3本柱で検証するのがよいです。

### ① ケースベース手動評価

代表ケースを作り、人間が結果をレビューする

### ② モジュール単位の妥当性確認

User Meaning、Item Meaning、Ranking などを分解して確認する

### ③ 再現性確認

同じ入力・同じ version なら同じ推薦理由を追えるか確認する

# 6. Phase1 の具体的な検証方法

---

## 6-1. ケースベース評価

## 目的

実際の贈答文脈に対して、推薦が自然かを確認する。

## 方法

代表ケースを複数作り、各ケースについて次を人間が確認する。

- user feature は妥当か
- 上位候補は妥当か
- NG商品が混ざっていないか
- 無難さ / 特別感のバランスは適切か

## ケース例の粒度

最低でも以下の軸をカバーしたいです。

| 軸           | 例                                          |
| ------------ | ------------------------------------------- |
| relationship | lover / boss / friend_close / family_parent |
| occasion     | birthday / thanks / apology / wedding_gift  |
| リスク志向   | 無難寄り / 特別寄り                         |
| 価格帯       | 低 / 中 / 高                                |
| 制約あり     | NG条件あり / なし                           |

## 推論

Phase1 では、網羅的な大量ケースより、**性格の異なる代表ケースを少数精鋭で作る** ほうがよいです。

---

## 6-2. モジュール単位レビュー

### A. User Meaning Estimation

確認したいこと:

- relationship / occasion の反映が妥当か
- 好み特徴・避けたい特徴が feature に適切に効いているか
- `λ_ctx` が直感に合うか

### B. Item Meaning Extraction

確認したいこと:

- 商品テキストから出た feature が違和感ないか
- conceptスコアが妥当か
- safety や novelty の過大評価がないか

### C. Matching / Ranking

確認したいこと:

- user と item の相性が直感に近いか
- popularity / risk の効き方が極端すぎないか

---

## 6-3. NG混入チェック

## 目的

MVPで最も避けたい「明らかな外れ」を検知する。

## 典型NG

- 上司向けでカジュアルすぎる
- お見舞いで派手すぎる
- お詫びで遊び心が強すぎる
- 予算外商品が出る
- 明示NG特徴の商品が出る

## 推論

Phase1 は「最高の推薦」より、まず **致命的な誤推薦を減らす** ことが重要です。

---

## 6-4. 再現性チェック

## 目的

後から「なぜこの結果になったか」を追えることを確認する。

## 確認内容

- 入力条件が残っているか
- user feature が残っているか
- item meaning version が分かるか
- final_score の内訳を追えるか

# 7. Phase1 で使う評価観点

ここは重要なので、評価軸として明示します。

| 評価観点ID | 評価観点   | 説明                             |
| ---------- | ---------- | -------------------------------- |
| E1         | 文脈適合性 | relationship / occasion に合うか |
| E2         | 安全性     | 失礼・不適切・外しすぎがないか   |
| E3         | 特別感     | 特別さが必要なケースで十分か     |
| E4         | 納得感     | なぜその商品なのか理解できるか   |
| E5         | 制約遵守   | 予算・NG条件を守るか             |
| E6         | 多様性     | 上位候補が同質すぎないか         |
| E7         | 一貫性     | 同種ケースで極端なブレがないか   |

## 推論

Phase1 では、特に **E1, E2, E5, E4** を重く見るのがよいです。

理由は、MVPでまず必要なのは「外さないこと」と「説明できること」だからです。

# 8. Phase1 の評価シート案

Notionやスプレッドシートで使いやすい形にすると、次の列があるとよいです。

| Case ID | Relationship | Occasion | Input Summary | Expected Direction | Recommended Items | Reviewer Score | NG有無 | コメント |
| ------- | ------------ | -------- | ------------- | ------------------ | ----------------- | -------------- | ------ | -------- |

### `Expected Direction` の例

- 無難・上品
- 安全寄りだが少し特別
- 温かみ重視
- 実用性高め
- 特別感強め

### `Reviewer Score` の例

5段階でもよいですが、Phase1 は細かすぎない方がよいです。

- 3 = 妥当
- 2 = やや違和感
- 1 = 明確に不適切

# 9. Phase1 で最低限保存すべき情報

ここが後のDB設計につながります。

| 保存対象                     | 必須度     | 理由                   |
| ---------------------------- | ---------- | ---------------------- |
| 正規化済み入力               | 必須       | 再現性                 |
| user feature                 | 必須       | user meaning妥当性確認 |
| λ_ctx                        | 必須       | 無難/特別の検証        |
| item meaning version         | 必須       | 再計算・比較           |
| candidate集合                | 必須       | retrieval妥当性確認    |
| feature match結果            | 必須       | matching妥当性確認     |
| final_score内訳              | 必須       | ranking妥当性確認      |
| 推薦結果一覧                 | 必須       | 手動評価               |
| evidence / explanation元情報 | あると良い | 説明可能性             |

## 推論

ここで大事なのは、**最終結果だけ保存しても不十分** という点です。

少なくとも Phase1 でも、途中の

- user feature
- candidate集合
- final_score内訳

は取ったほうがよいです。

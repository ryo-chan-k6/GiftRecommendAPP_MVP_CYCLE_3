# 1. このドキュメントの目的

本ドキュメントは、Gift Recommendation Service の全体アーキテクチャを構成する各モジュールについて、以下を整理することを目的とする。

- モジュールの責務
- 主な入出力
- オンライン処理かバッチ処理か
- MVPで実装するか、後続フェーズで実装するか
- 後続のテスト/検証設計・データ設計で必要となる観点

---

# 3. モジュール詳細定義

---

## M01. User Context / Intent Input

### 役割

ユーザーが入力した贈答条件を受け取り、後続処理で使いやすい構造に整形する。

### 主な入力

- relationship
- occasion
- 好みの特徴
- 避けたい特徴
- 予算
- NG条件

### 主な出力

- 正規化済み入力データ
- 欠損補完済みのレコメンド条件

### 処理概要

- 入力値のバリデーション
- relationship / occasion のコード化
- 予算範囲の正規化
- NG条件の構造化

### MVPでの扱い

**対象**

### 補足

ここではまだ意味推定は行わず、あくまで入力受付と整形を担当する。

---

## M02. User Meaning Estimation

### 役割

ユーザー入力から、推薦計算に使う user feature を推定する。

### 主な入力

- M01の正規化入力
- relationship rule
- occasion rule
- pair rule
- hint dictionary / semantic concept

### 主な出力

- `integrated_feature`
- `user_social`
- `user_symbolic`
- `λ_ctx`

### 内部ロジック

- relationship / occasion による基本feature付与
- pair rule による補正
- hint / semantic concept による補正
- social / symbolic 集約
- λ_ctx 算出

### MVPでの扱い

**対象**

### 補足

本サービスの心理モデル側の中核モジュール。

---

## M03. Hard Filter

### 役割

意味一致度を見る前に、明示条件で商品候補を除外する。

### 主な入力

- 正規化済みレコメンド条件
- 商品マスタ
- 商品属性

### 主な出力

- フィルタ後候補集合

### フィルタ例

- 予算外
- 明示NGジャンル
- 明示NG特徴（タグ）

### MVPでの扱い

**対象**

### 補足

ここは精度向上というより、明確な不適合を除外するための処理。

---

## M04. Candidate Retrieval

### 役割

全商品から、意味的に近い候補商品を粗く抽出する。

### 主な入力

- user meaning
- item meaning
- semantic retrieval用特徴量

### 主な出力

- 上位N件の候補商品ID集合

### 実装候補

- embedding 類似検索
- meaning空間近傍検索
- conceptベース粗検索

### MVPでの扱い

**対象**

### 補足

全件に対して精密スコアを計算しないための前段候補生成。

---

## M05. Meaning Matching

### 役割

候補商品について、user feature と item feature の一致度を詳細計算する。

### 主な入力

- user feature
- candidate items
- item feature

### 主な出力

- featureごとの match
- social_match
- symbolic_match
- context_score の元データ

### 計算例

- `distance_f = |user_f - item_f|`
- `match_f = 1 - distance_f`
- `social_match = mean(...)`
- `symbolic_match = mean(...)`

### MVPでの扱い

**対象**

### 補足

推薦妥当性の中核。

「商品の性質」ではなく「ユーザーとの適合」を計算するモジュール。

---

## M06. Final Ranking

### 役割

Matching結果に popularity / risk / λ_ctx を加味し、最終順位を決定する。

### 主な入力

- context_score
- popularity signals
- risk signals
- λ_ctx

### 主な出力

- `final_score`
- 商品順位

### 主な要素

- review_rating
- review_count
- ranking
- niche性
- 個性の強さ
- λ_ctx による重み調整

### MVPでの扱い

**対象**

### 補足

無難寄りと特別寄りのバランスを最終的に調整するモジュール。

---

## M07. Recommendation Result Builder

### 役割

ランキング済み商品を、フロントへ返すレスポンス形式に整形する。

### 主な入力

- 商品順位
- 商品情報
- 付随スコア

### 主な出力

- 推薦商品一覧
- 商品表示用データ
- 必要に応じて説明メタデータ

### MVPでの扱い

**対象**

### 補足

APIレスポンスや表示要件に合わせて整形する層。

---

## M08. Explanation Generation

### 役割

推薦理由を説明文として生成する。

### 主な入力

- item feature
- user feature
- match結果
- evidence
- context

### 主な出力

- 推薦理由文
- 簡易ラベル
- 説明メタデータ

### MVPでの扱い

**簡易対象**

### MVP案

### 将来拡張

- LLMを使った自然文説明
- 根拠付き説明生成

---

### 役割

推薦結果に対するユーザーの行動ログを記録する。

### 主な入力

# 10. ここまでの意味

## 事実

T1で全体アーキテクチャを整理し、T2でモジュールと機能を分解しました。

## 推論

この状態になると、次にやるべきはかなり明確です。

次のステップでは、**どのフェーズで何を実装・検証するか** を切ることで、

- 何を先に作るか
- 何を後回しにするか
- どのデータをMVPで保存すべきか

が見えてきます。

---

次はこのまま **T3：MVP〜最終系ロードマップ** を、

Notionでそのまま管理できる形で整理するのが自然です。

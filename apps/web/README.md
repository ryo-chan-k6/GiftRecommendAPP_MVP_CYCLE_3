# apps/web

## 1. 役割

`apps/web` は Gift Recommendation Service のフロントエンドアプリケーションである。

主な責務は以下とする。

- 画面表示
- ユーザー入力受付
- 画面遷移
- API 呼び出し
- 画面状態管理
- エラー表示
- ローディング / 入力制御などのUI非機能対応

---

## 2. 責務境界

### 実施してよいこと

- UIコンポーネントの実装
- 画面イベントの処理
- APIへのリクエスト送信
- 画面状態・入力状態の管理
- API結果の表示
- 例外ケース一覧に基づくエラー表示

### 実施してはいけないこと

- Recoロジックの実装
- DBへの直接アクセス
- Batch都合の処理の実装
- API内部ロジックの再実装
- 画面設計書にない重要仕様の独自追加

---

## 3. 参照すべき設計書

UI実装前に、少なくとも以下を参照すること。

- `docs/.../画面設計書.md`
- `docs/.../画面イベント一覧.md`
- `docs/.../例外ケース一覧.md`
- `docs/.../画面非機能観点整理表.md`
- `docs/.../API仕様書.md`
- `docs/.../認証認可方針書.md`（必要に応じて）

---

## 5. ディレクトリ責務

| ディレクトリ     | 役割                       |
| ---------------- | -------------------------- |
| `src/app`        | ルーティング、画面エントリ |
| `src/components` | 汎用UI部品                 |
| `src/features`   | 画面・業務単位のUIロジック |
| `src/hooks`      | UI補助ロジック             |
| `src/services`   | API呼び出し                |
| `src/lib`        | 汎用関数                   |
| `src/styles`     | スタイル                   |
| `src/types`      | Web側で利用する型          |

---

## 6. 実装ルール

- docs を正本として実装する
- 画面設計書・イベント一覧・例外ケース一覧との整合を取る
- API仕様書にない前提で実装しない
- 主要操作は追跡可能性を意識する
- UI都合で API / Reco の仕様を暗黙変更しない

---

## 7. テスト観点

最低限、以下を確認すること。

- 主要導線が成立する
- 主要イベントが動作する
- API連携が成立する
- 代表例外ケースが表示される
- ローディング / 二重送信防止が成立する

---

## 8. 補足

このディレクトリは「見た目を作る場所」ではなく、

**画面仕様・イベント・例外・非機能を成立させる場所** として扱うこと。
